二、漏洞分析与修复

Vulnerability Analysis & Remediation

1

IDOR 越权查看他人资料

严重 · CVSS 8.6

CWE-639 · 个人中心 GET /profile?user_id=N

问题：user_id 从 URL 参数获取，用户修改 URL 即可访问任意用户的资料，无归属验证。

Python — 问题代码
user_id = request.args.get("user_id", "")    # 从 URL 参数获取 — 用户可控！
sql = f"SELECT ... FROM users WHERE id = {user_id}"
c.execute(sql)

攻击复现：alice 登录后，修改 ?user_id=1 即可查看 admin 的邮箱、手机、余额等信息。

❌ 修复前request.args.get("user_id") 用户可控
✅ 修复后get_current_user_id() 从 Session 获取
2

IDOR 越权为他人充值

严重 · CVSS 9.0

CWE-639 · 充值接口 POST /recharge

问题：充值操作的 user_id 来自表单，攻击者可修改为其他用户 ID。

# alice 给 admin 充值
curl -b /tmp/alice.txt -X POST http://target/recharge \
  -d "user_id=1&amount=100"

# 结合负值从 admin 账户扣钱
curl -b /tmp/alice.txt -X POST http://target/recharge \
  -d "user_id=1&amount=-10000"
❌ 修复前表单 user_id 不校验归属
✅ 修复后与 Session ID 比对，不一致则 403
3

负值金额充值

高危 · CVSS 7.5

CWE-841 · 充值接口 POST /recharge

问题：未对 amount 做正负校验，amount=-200 可使余额减少。

SQL 语义分析
正常充值 amount=100:   UPDATE users SET balance = balance + 100    → 余额增加 100
恶意充值 amount=-200:  UPDATE users SET balance = balance + (-200) → 余额减少 200
❌ 修复前输入 -200 余额减少
✅ 修复后if amount <= 0: reject
4

超额充值

高危 · CVSS 7.0

无上限校验，输入极大值可暴增余额。

# Alice 一次充值 999999 元
curl -b /tmp/alice.txt -X POST http://target/recharge \
  -d "user_id=2&amount=999999"
# 余额变为 1000999
合理限制：单次充值上限（如10万元）· 单日累计上限（如50万元）· 账户余额上限 · 频率限制
5

无上限重复充值

中危

无防重放机制，可批量重复提交刷余额。

for i in {1..100}; do
  curl -b /tmp/alice.txt -X POST http://target/recharge \
    -d "user_id=2&amount=1000" &
done
# 余额增加 1000 × 100 = 100000
❌ 修复前无防重放，可无限次提交
✅ 修复后充值后刷新 CSRF Token，一次性使用
6

SQL 注入（profile + recharge）

严重 · CVSS 9.8

两处接口均使用 f-string 拼接 SQL。

Python — 两处 f-string 注入
# 个人中心 — SELECT 注入
sql = f"SELECT ... FROM users WHERE id = {user_id}"

# 充值 — UPDATE 注入
sql = f"UPDATE users SET balance = balance + {amount} WHERE id = {user_id}"
❌ 修复前f"WHERE id = {user_id}"
✅ 修复后"WHERE id = ?" 参数化查询
7

CSRF 跨站充值请求

中危

CWE-352 · 充值接口无 CSRF Token 校验

攻击者构造恶意页面： <form action="http://target/recharge" method="POST" style="display:none"> <input name="user_id" value="1"> <input name="amount" value="10000"> </form> <script>document.forms[0].submit()</script> 当已登录用户访问该页面 → 浏览器自动附带 Cookie → 余额被篡改
❌ 修复前无 CSRF 校验
✅ 修复后validate_csrf() Token 校验
8

重放攻击与重复提交

中危

CSRF Token 通过后未刷新，同一 Token 可反复使用。

❌ 修复前Token 可重复使用
✅ 修复后refresh_csrf() 一次性 Token

三、Burp Suite 攻击复现

Attack Reproduction

3.1 IDOR 越权查看

步骤1：以 alice 身份登录 → 点击「个人中心」
Burp 拦截到：GET /profile?user_id=2

步骤2：发送到 Repeater（Ctrl+R）
user_id=1 → 查看 admin 越权成功 → 返回管理员资料
user_id=3 → 查看新注册用户 越权成功 → 返回用户资料

步骤3：使用 Intruder 批量枚举
GET /profile?user_id=§1§ → Payload: Numbers 1-100 → 找出所有有效用户

3.2 负值充值测试

拦截充值请求：
POST /recharge HTTP/1.1
user_id=2&amount=100
↓ 修改为
user_id=2&amount=-99999

充值前余额：100 → 充值后余额：-99899（负值充值成功！）

四、业务逻辑漏洞防护方案

Security Hardening

4.1 权限校验五层模型

客户端请求 │ ▼ ┌─────────────────────────┐ │ 1. 身份认证 │ ← @login_required │ （是否已登录？） │ └────────┬────────────────┘ │ ┌────────▼────────────────┐ │ 2. 对象归属验证 │ ← 从 Session 获取，不从参数获取 │ （水平越权防护） │ 防止 IDOR 越权 └────────┬────────────────┘ │ ┌────────▼────────────────┐ │ 3. 业务参数校验 │ ← amount > 0, amount < 上限 │ （逻辑漏洞防护） │ 防止负值/超额 └────────┬────────────────┘ │ ┌────────▼────────────────┐ │ 4. CSRF Token 校验 │ ← validate_csrf() │ （跨站防护） │ 防止跨站伪造 └────────┬────────────────┘ │ ┌────────▼────────────────┐ │ 5. 防重放 │ ← refresh_csrf() │ （重复提交防护） │ 一次性 Token └────────┬────────────────┘ │ ▼ 正常执行业务逻辑

4.2 核心修复代码

Python — 完整修复
@app.route("/profile")
@login_required
def profile():
    current_user_id = get_current_user_id()  # 从 Session 获取
    if current_user_id is not None:
        c.execute(
            "SELECT id, username, email, phone, balance FROM users WHERE id = ?",
            (current_user_id,),
        )
        row = c.fetchone()
    return render_template("profile.html", user=user_data, csrf_token=..., error=error)


@app.route("/recharge", methods=["POST"])
@login_required
def recharge():
    # 1. CSRF 校验
    if not validate_csrf():
        abort(403, "CSRF 验证失败")

    # 2. IDOR 校验：只能操作自己的账户
    current_id = get_current_user_id()
    if not current_id or str(current_id) != str(user_id):
        abort(403, "无权操作其他用户的账户")

    # 3. 金额正数校验
    if amount <= 0:
        return redirect("/profile?error=充值金额必须大于0")

    # 4. 金额上限校验
    if amount > 100000:
        return redirect("/profile?error=单次充值金额不能超过10万元")

    # 5. 参数化查询
    c.execute("UPDATE users SET balance = balance + ? WHERE id = ?", (amount, user_id))

    # 6. 刷新 CSRF Token（防重放）
    refresh_csrf()
    return redirect("/profile")

五、总结与反思

Summary & Reflections

5.1 三大铁律

铁律1：永远从 Session 获取当前用户身份，不信任客户端参数 铁律2：所有业务参数必须在服务端做范围校验（正/负、上限/下限） 铁律3：幂等操作必须防重放（一次性 Token + 唯一请求标识）

5.2 IDOR 三种防御层次

最佳方案：完全避免从客户端获取对象标识符 → 从 Session 推导用户身份 次优方案：验证对象归属 → 比较参数中的 ID 是否与当前用户匹配 兜底方案：使用不可预测的标识符 → UUID 替代自增 ID（但不能替代归属验证）

5.3 8个漏洞回顾

#漏洞路由严重攻击方式修复关键
V1IDOR 越权查看/profile严重修改 ?user_id=N从 Session 获取 ID
V2IDOR 越权充值/recharge严重修改表单 user_id比对 Session ID
V3负值充值/recharge高危输入负金额amount <= 0
V4超额充值/recharge高危输入极大金额amount > 100000
V5重复提交/recharge中危重复发送请求刷新 CSRF Token
V6SQL 注入两处接口严重注入 payload参数化查询
V7CSRF 跨站/recharge中危跨站提交表单validate_csrf()
V8重放攻击/recharge中危多次发送拦截请求一次性 Token

业务逻辑漏洞是所有安全漏洞中最"人性化"的一种。

它不是因程序员忘了加过滤，而是因程序员默认用户会按规矩操作。但安全开发的核心思维恰恰是：永远假设用户会尝试做任何你能想象到的坏事。

5.4 后续学习方向

方向说明难度
竞争条件并发请求导致数据不一致（如重复提现）进阶
JWT 安全Token 伪造和签名算法混淆进阶
OAuth 2.0 安全授权码劫持、CSRF、重定向 URI 篡改高阶
支付逻辑金额篡改、数量篡改、订单替换高阶

5.5 安全检查清单

认证与会话管理： [ ] 敏感操作是否从 Session 获取用户身份，而非客户端参数？ [ ] 是否验证了操作对象属于当前用户？ [ ] Session 是否配置了 HttpOnly + SameSite？ 业务参数校验： [ ] 金额、数量等数值是否做了正数校验？ [ ] 是否设置了合理的上限和下限？ [ ] 是否校验了数据类型？ [ ] 是否校验了前端没有限制的参数？ 防篡改与防重放： [ ] 关键业务操作是否有 CSRF 防护？ [ ] CSRF Token 是否一次性使用？ [ ] 是否有操作频率限制？
