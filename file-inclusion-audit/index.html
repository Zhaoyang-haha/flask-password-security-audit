1.2 文件包含 vs 路径遍历

文件包含漏洞与路径遍历漏洞经常同时出现： 路径遍历：通过 ../ 等序列访问不在预期目录中的文件 例如：/page?name=../../../etc/passwd 本地文件包含：将文件内容包含到页面输出中 例如：/page?name=../app.py → app.py 内容被渲染到页面 在本项目中，/page?name= 同时存在这两个漏洞： · 路径遍历：通过 ../ 穿越到 pages/ 目录之外 · 文件包含：遍历到的文件内容被读取并显示在页面中

1.3 功能缺陷汇总

缺陷严重说明
无路径校验（直接拼接）严重os.path.join 直接拼接，../ 可穿越目录
无字符过滤严重.、/ 等路径穿越字符未被过滤
无路径规范化验证严重未使用 os.path.realpath 校验
无访问控制中危未登录也可访问
输出使用 | safe中危文件内容直接渲染，存在 XSS 风险

二、漏洞分析与修复

Vulnerability Analysis & Remediation

1

本地文件包含 LFI / 路径遍历

严重 · CVSS 9.0

CWE-22 · 动态页面加载 GET /page?name=

Python — 问题代码
name = request.args.get("name", "")
file_path = os.path.join(PAGES_DIR, name)  # 直接拼接用户输入！
if os.path.isfile(file_path):
    with open(file_path, "r") as f:
        page_content = f.read()

攻击复现：读取源码 ?name=../app.py、系统文件 ?name=../../../../etc/passwd、数据库 ?name=../data/users.db，全部成功！

❌ 修复前os.path.join(PAGES_DIR, name) 用户可控
✅ 修复后字符过滤 + 路径规范化 + 目录白名单
2

远程文件包含 RFI

严重

理论风险：虽然 open() 不支持 URL，但如果代码改为 urllib/requests 库则存在 RFI 风险。

防护：只允许本地文件包含，拒绝 URL 协议；限制可访问的文件目录范围。
3

文件内容泄露（敏感信息暴露）

高危

CWE-200 · 被包含的文件内容直接显示在页面中。

泄露路径泄露内容危害
../app.pyapp.secret_key、DB_PATH、路由逻辑严重
../data/users.db所有用户数据、密码哈希高危
../../../../etc/passwd系统用户列表中危
4

XSS 通过文件内容渲染

中危

CWE-79 · {{ page_content | safe }} 跳过 HTML 转义

如果 page_content 中包含 <script> 标签，会被浏览器直接执行。

防护：将 | safe 替换为普通输出 {{ page_content }}（自动转义）；或确保来源可信。
5

无认证访问动态页面

中危

CWE-862 · 未登录也可访问

❌ 修复前无 @login_required
✅ 修复后根据业务需要添加认证

三、Burp Suite 攻击复现

Attack Reproduction

3.1 路径遍历读取文件

步骤1：访问 /page?name=help → Burp 拦截
步骤2：发送到 Repeater（Ctrl+R）

测试1：读取应用源码
GET /page?name=../app.py HTTP/1.1
→ 响应中包含 app.py 的源代码

测试2：读取系统文件
GET /page?name=../../../../etc/passwd HTTP/1.1
→ 响应中包含 /etc/passwd 的内容

测试3：URL编码绕过
GET /page?name=%2E%2E%2F%2E%2E%2Fapp.py HTTP/1.1
→ 响应中包含 app.py 源代码

3.2 Intruder 批量探测

将 name 设置为 payload 位置：GET /page?name=§help§

Payload 列表：
../app.py
../../../../etc/passwd
../data/users.db
....//....//....//etc/passwd
%2e%2e%2f%2e%2e%2fapp.py

攻击方式：Sniper → 观察响应长度，异常的可能文件包含成功

四、防护方案

Security Hardening

4.1 四层防护体系

文件包含漏洞防护体系 第一层：输入字符过滤（基础防御） re.sub(r"[^a-zA-Z0-9_\-]", "", name) 只允许字母、数字、下划线、短横线 → 彻底阻断 ../ 等路径穿越字符 第二层：路径规范化验证（核心防御） os.path.realpath() 获取真实路径 str.startswith(pages_real) 验证在 pages 目录内 → 防止经过编码/变形的路径穿越 第三层：合理的输出安全（辅助防御） 不建议在公开页面使用 | safe → 降低 XSS 风险 第四层：错误信息保护（辅助防御） 统一显示"页面不存在" 不泄露文件路径、不存在的原因等 → 防止信息泄露

4.2 核心修复代码

Python — 完整修复
@app.route("/page")
def dynamic_page():
    name = request.args.get("name", "")
    page_content = None
    if name:
        # 第一层：字符白名单过滤
        safe_name = re.sub(r"[^a-zA-Z0-9_\-]", "", name)
        if not safe_name:
            page_content = "页面不存在"
        else:
            # 第二层：路径规范化与目录验证
            file_path = os.path.join(PAGES_DIR, safe_name)
            real_path = os.path.realpath(file_path)
            pages_real = os.path.realpath(PAGES_DIR)

            if not real_path.startswith(pages_real + os.sep) \
               and real_path != pages_real:
                page_content = "页面不存在"
            elif os.path.isfile(real_path):
                try:
                    with open(real_path, "r", encoding="utf-8") as f:
                        page_content = f.read()
                except Exception:
                    page_content = None
            else:
                # 尝试加 .html 后缀
                file_path_html = os.path.join(PAGES_DIR, safe_name + ".html")
                real_path_html = os.path.realpath(file_path_html)
                if not real_path_html.startswith(pages_real + os.sep):
                    page_content = "页面不存在"
                elif os.path.isfile(real_path_html):
                    with open(real_path_html, "r") as f:
                        page_content = f.read()
                else:
                    page_content = None
        if page_content is None:
            page_content = "页面不存在"
    return render_template("index.html", page_content=page_content)

4.3 防御原理解析

第一阶段：字符过滤 输入："../../../etc/passwd" 过滤：re.sub(r"[^a-zA-Z0-9_\-]", "", name) 结果："----------etcpasswd"（特殊字符全被移除） → 路径穿越字符被彻底清除！ 第二阶段：路径规范化 os.path.realpath() 将相对路径转为绝对路径 验证是否以 PAGES_DIR 开头 第三阶段：双层防御 · 字符过滤 + 路径验证 = 双层保险 · 任何一层都能拦截攻击

安全机制修复前修复后

"一个字符的信任，就可能让整个系统沦陷。"

文件包含漏洞只需要一个简单的 ../，就能让攻击者读取系统中任何可读的文件。防御的方式也同样简单：永远不要相信用户输入的路径，永远不要直接将用户输入拼接到文件路径中。

从第一轮的密码安全加固，到最后一轮的文件包含防护，本项目累计修复 29 个漏洞，覆盖 OWASP Top 10 中的大部分类别。安全不是最终的状态，而是持续的过程。
