Project Overview

三、原始漏洞分析

Vulnerability Analysis

通过对项目源代码的逐行审计，我们发现了以下 7 项 安全漏洞，按照严重程度排列如下：

1

密码明文存储

严重

问题代码（app.py）：

Python 
USERS = {
    "admin": {
        "username": "admin",
        "password": "admin123",      # ← 明文存储！
        "role": "admin",
        "email": "admin@example.com",
        "phone": "13800138000",
        "balance": 99999
    },
    "alice": {
        "username": "alice",
        "password": "alice2025",     # ← 明文存储！
        # ...
    }
}

危害分析： 如果数据库文件被泄露（或攻击者通过其他漏洞获取了 USERS 字典的内容），所有用户的密码将以可读形式直接暴露。这是 Web 应用安全中最基础也是最严重的违规行为之一，违反了 OWASP Top 10 中关于 "敏感数据暴露"（A02:2021 – Cryptographic Failures）的安全要求。

验证方式： 直接查看源代码即可看到所有账号的明文密码。

OWASP 映射： A02:2021 – Cryptographic Failures

2

密码明文比对

严重

问题代码（app.py）：

Python 
if username in USERS and USERS[username]["password"] == password:

危害分析： 使用 == 直接进行字符串比对，存在以下问题：

  • 意味着密码必须明文存储才能比对，与漏洞 1 形成耦合风险
  • 无法防御 时序攻击（Timing Attack）——攻击者可以通过响应时间的细微差异逐字符猜测密码
  • 不兼容哈希存储方案，无法迁移到安全的密码存储方式

推荐做法： 使用 check_password_hash() 进行常量时间比较（Constant-Time Comparison）。

3

密码显示在页面上

高危

问题代码（templates/index.html）：

HTML 
<!-- 密码直接显示在页面上 -->
<li><span class="info-label">密码：</span>{{ user_info["password"] }}</li>

危害分析： 登录成功后，用户的密码直接渲染在网页上。如果用户离开电脑未锁屏，路过的人可直接看到密码。这违反了 最小权限原则（Principle of Least Privilege）——前端页面不应展示任何不必要的信息。此外，如果页面被截图或共享，密码也会随之泄露。

实际验证： 登录后的页面上直接显示 "密码：admin123"。
4

HTML 注释泄露默认账号

高危

问题代码（templates/login.html）：

HTML 
<!-- 调试信息 - 默认管理员账号 用户名: admin 密码: admin123 -->

危害分析： 这是开发阶段的调试残留，存在严重的安全隐患：

  • 任何查看网页源代码（浏览器右键 → 查看页面源代码）的人都能直接获得管理员账号密码
  • 攻击者不需要进行任何密码爆破或猜测即可获得管理员权限

攻击链路： 访问登录页面 → 查看 HTML 源代码 → 直接获取管理员凭证 → 完全控制系统。

5

弱 Secret Key

中危

问题代码（app.py）：

Python 
app.secret_key = "dev-key-2025"   # ← 硬编码弱密钥！

危害分析： secret_key 用于 Flask 签名 session cookie。使用硬编码的弱密钥，攻击者可以：

  • 伪造任意用户的 session cookie，实现 会话劫持（Session Hijacking）
  • 通过已知密钥解密 session 数据，获取敏感信息
  • 如果应用存在反序列化漏洞，可导致远程代码执行（RCE）

OWASP 映射： A07:2021 – Identification and Authentication Failures

6

Debug 模式开启

中危

问题代码（app.py）：

Python 
app.run(debug=True, host="0.0.0.0", port=5000)

危害分析： Flask 的 debug 模式会在代码出错时显示详细的调试页面，包含：

  • 完整的源代码片段
  • 所有环境变量（可能包含数据库密码、API 密钥等）
  • 当前请求的各种变量值
  • Python 交互式控制台（可远程执行代码）

生产环境中开启 debug 模式是 极其危险的配置错误。

OWASP 映射： A05:2021 – Security Misconfiguration

7

无爆破防护

中危

危害分析： 登录接口没有设置失败次数限制，攻击者可以使用 Hydra、Medusa 等自动化工具对账号进行 无限次密码爆破，结合弱口令字典（如 rockyou.txt），成功率极高。

典型攻击命令：

Bash 
hydra -l admin -P /usr/share/wordlists/rockyou.txt \
  192.168.14.129 http-post-form \
  "/login:username=^USER^&password=^PASS^:登录失败"

OWASP 映射： A07:2021 – Identification and Authentication Failures

四、Burp Suite 抓包验证

Burp Suite Packet Capture & Verification

4.1 抓包过程

为了验证密码在 HTTP 传输过程中的安全性，我们使用 Burp Suite 进行了中间人代理抓包测试。具体步骤如下：

  1. 配置 Burp Suite 代理监听 127.0.0.1:8080
  2. 浏览器设置 HTTP 代理为 127.0.0.1:8080
  3. 访问 http://192.168.14.129:5000/login
  4. 输入用户名 admin，密码 admin123 并提交登录表单
  5. Burp Suite 成功拦截到 POST 请求

4.2 拦截内容分析

Burp Suite 的 Proxy → Intercept 面板中捕获到的 HTTP 请求如下：

POST /login HTTP/1.1
Host: 192.168.14.129:5000
User-Agent: Mozilla/5.0 (X11; Linux x86_64)
Accept: text/html,application/xhtml+xml
Content-Type: application/x-www-form-urlencoded
Content-Length: 37

username=admin&password=admin123 ← 密码明文传输
🚨 发现 username=admin&password=admin123 以明文形式出现在 HTTP 请求体中。在没有 HTTPS 加密的情况下，任何能截获网络流量的人（如同 WiFi 网络中的其他用户、ISP、中间路由器）都可以直接读取登录凭据。

4.3 HTTP vs HTTPS 对比

五、修复方案

Remediation Plan

针对上述 7 项安全漏洞，我们逐一制定了修复方案，并实施了代码层面的修改。以下详述每项修复的思路、具体代码变更及修复效果。

5.1 Werkzeug 密码哈希存储

修复思路： 不存储明文密码，而是存储密码的哈希值。采用 Werkzeug 库提供的 generate_password_hash() 函数，该函数默认使用 PBKDF2-HMAC-SHA256 算法，迭代次数为 600,000 次，并自动添加随机盐值（Salt），有效抵御彩虹表攻击（Rainbow Table Attack）和 GPU 并行破解。

修复后代码（app.py）：

Python 
from werkzeug.security import generate_password_hash, check_password_hash

USERS = {
    "admin": {
        "username": "admin",
        "password": generate_password_hash("admin123"),  # ← 哈希存储
        "role": "admin",
        "email": "admin@example.com",
        "phone": "13800138000",
        "balance": 99999
    },
    "alice": {
        "username": "alice",
        "password": generate_password_hash("alice2025"), # ← 哈希存储
        # ...
    }
}
💡 效果说明 数据库中存储的是类似 pbkdf2:sha256:600000$<salt>$<hash> 的哈希值，即使数据泄露也无法还原出原始密码。每次调用 generate_password_hash() 会生成不同的盐值，因此相同密码的哈希结果也不同。 哈希算法演进建议：werkzeug → bcrypt → argon2（当前最推荐的自适应哈希算法）。

5.2 安全密码验证

修复思路： 使用 Werkzeug 提供的 check_password_hash() 函数替代直接字符串比对。该函数内置了 常量时间比较（Constant-Time Comparison），有效防御时序攻击。

❌ 修复前 if USERS[username]["password"] == password:
✅ 修复后 if check_password_hash(USERS[username]["password"], password):

5.3 不在模板中输出密码

修复思路： 从根本上消除密码在页面上的显示，采取双重保障措施：① 在模板中删除密码展示行；② 在后端传递给模板前过滤掉密码字段。

修复后代码（templates/index.html）：

HTML 
<!-- 修复前 -->
<li><span class="info-label">密码：</span>{{ user_info["password"] }}</li>

<!-- 修复后 -->  ← 整行删除，不再显示密码

后端配合修改（app.py）：

Python 
# 传递给模板前过滤掉密码字段
user_info = {k: v for k, v in USERS[username].items() if k != "password"}

5.4 删除泄露信息的 HTML 注释

修复思路： 移除包含默认账号密码的调试注释。对于需要记录此类信息的场景，应使用环境变量或独立的配置文件管理，并将其添加到 .gitignore 中。

❌ 修复前 <!-- 调试信息 - 默认管理员账号 用户名: admin 密码: admin123 -->
✅ 修复后 <!-- 整行删除 -->

5.5 随机 Secret Key

修复思路： 使用 os.urandom(32) 在每次应用启动时生成 32 字节（256 位）的随机密钥，替代硬编码的弱密钥。该密钥源自操作系统提供的加密安全随机数生成器（CSPRNG）。

❌ 修复前 app.secret_key = "dev-key-2025"
✅ 修复后 app.secret_key = os.urandom(32) # 256-bit random key
💡 进阶建议 对于生产环境，建议将 secret_key 存储于环境变量中（如 os.environ.get('FLASK_SECRET_KEY')），确保应用重启后 session 仍可解密，同时避免密钥硬编码在代码库中。

5.6 关闭 Debug 模式

修复思路： 将 debug=True 改为 debug=False，并建议将 debug 模式的控制权交给环境变量，实现开发与生产环境的自动切换。

❌ 修复前 app.run(debug=True, host="0.0.0.0", port=5000)
✅ 修复后 app.run(debug=False, host="0.0.0.0", port=5000)
💡 推荐做法 debug = os.environ.get('FLASK_ENV') == 'development'
app.run(debug=debug, host="0.0.0.0", port=5000)

5.7 爆破防护

修复思路： 实现基于 IP 的登录失败计数与临时锁定机制。同一 IP 地址在 60 秒内连续输错 3 次密码后，将被临时锁定 60 秒，大幅增加自动化爆破的时间成本。

修复后代码（app.py）：

Python 
# 登录失败计数（内存存储）
login_attempts = {}

# 在登录验证中检查
client_ip = request.remote_addr
if client_ip in login_attempts:
    attempts, lock_time = login_attempts[client_ip]
    if attempts >= 3 and (time.time() - lock_time) < 60:
        error = "登录失败次数过多，请60秒后再试。"
        return render_template("login.html", error=error)

# 登录失败后记录
login_attempts[client_ip] = (count + 1, now)

# 登录成功后清除计数
login_attempts.pop(client_ip, None)
💡 设计考量 本方案采用内存字典存储，适用于单进程部署。对于生产环境的分布式部署，建议改用 Redis 等集中式缓存存储失败计数，实现跨进程的锁定同步。更进一步的方案包括：reCAPTCHA、账户临时冻结、递增延迟（Exponential Backoff）等。

八、总结与反思

Summary & Reflections

8.1 安全开发五大原则

通过本次安全漏洞分析与修复实践，我们总结出以下五项 Web 安全开发的核心原则：

  • 永远不要存储明文密码 — 使用经过充分测试的哈希库（如 werkzeug.security、bcrypt、argon2），选择带有自适应迭代次数的算法（如 PBKDF2、bcrypt、argon2），以抵御暴力破解和硬件加速攻击（GPU/ASIC）。
  • 最小权限原则 — 不应在前端展示敏感字段（密码、密钥、身份证号等）。传递数据到模板前，务必使用字典推导、对象筛选等方式过滤掉不需要展示的敏感信息。
  • 不要信任注释 — 代码注释中的敏感信息同样是安全风险。在生产环境的代码中不应保留包含密码、密钥、连接字符串等敏感信息的注释。使用 .gitignore 管理配置文件。
  • 使用安全的默认配置 — debug=True 仅用于开发环境，上线前必须关闭。所有安全相关的配置项都应默认采用安全的设置，通过环境变量控制不同环境的差异。
  • 防御纵深（Defense in Depth） — 单一安全措施不足以提供充分保护。应采用多层防御：密码哈希防泄露、爆破防护防暴力破解、HTTPS 防中间人攻击、CSRF Token 防跨站请求伪造、Session 安全配置防劫持。

8.2 本次作业未修复项

由于实验环境的限制，以下安全风险本次未能完全解决，作为后续迭代的优化方向：

未修复项风险等级说明与建议
HTTP 明文传输 高危 由于在私有内网测试环境，未配置 HTTPS。生产环境必须使用 HTTPS + HSTS 防止中间人窃听。建议通过 Let's Encrypt（免费 TLS 证书）或 Nginx 反向代理实现。
Session 安全配置 中危 可进一步配置 session_cookie_secure=True（仅 HTTPS 传输）、session_cookie_httponly=True（禁止 JavaScript 访问）、session_cookie_samesite='Lax' 等安全选项。
CSRF 防护 中危 登录表单未添加 CSRF Token，可被跨站请求伪造攻击利用。建议集成 Flask-WTF 库统一管理 CSRF 防护，或手动在表单中嵌入 csrf_token。

8.3 安全审计方法论总结

本次审计遵循了系统化的安全评估流程：源代码审查（Static Analysis）→ 漏洞分类与风险评估 → 动态抓包验证（Dynamic Analysis）→ 修复实施 → 回归测试验证。这一方法论可复用于其他 Web 应用的安全审计工作。

"Security is not a product, but a process." — 安全不是一个产品，而是一个持续的过程。

真正的安全来自持续的关注、定期的审计和不断的改进，而非一次性的修复。
