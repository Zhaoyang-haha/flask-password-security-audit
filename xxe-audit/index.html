一、实验概述

Experiment Overview

1.1 什么是 XXE？

XXE（XML External Entity Injection） 是一种针对 XML 解析器的攻击，攻击者通过在 XML 中定义恶意外部实体，利用解析器在处理实体时的行为，实现文件读取、SSRF、DoS 等攻击。

攻击者提交 XML → 定义 ENTITY SYSTEM "file:///etc/passwd" → 解析器遇到 &xxe; → 查找 ENTITY 定义 → SYSTEM 路径 → 读取文件 → 内容替换 → 信息泄露！

二、漏洞分析与修复

Vulnerability Analysis & Remediation

1

XXE 外部实体读取任意文件

严重 · CVSS 9.0

CWE-611 · 正则提取 ENTITY 路径后 open() 读取任意文件

Python — 问题代码
entity_names = re.findall(r'<!ENTITY\s+(\w+)\s+SYSTEM\s+"([^"]+)"', xml_data)
for name, path in entity_names:
    if path.startswith("file://"):
        path = path[7:]
    with open(path, "r") as f:           # 读取任意文件！
        content = f.read()
    xml_data = xml_data.replace(f"&{name};", content)

攻击复现：提交含 <!ENTITY xxe SYSTEM "file:///etc/passwd"> 的 XML，即可读取系统密码文件、app.py 源码、数据库文件等。

2

XXE + SSRF 内网探测

严重

SYSTEM 指定 http:// URL 时可发起 SSRF

如果代码使用 urllib.urlopen() 或真正的 XML 解析器（如 lxml），则可通过 XXE 探测内网：http://127.0.0.1:5000/、http://169.254.169.254/（云元数据）等。

3

XXE DoS 拒绝服务（Billion Laughs）

高危

实体扩展呈指数增长，耗尽服务器内存/CPU

Billion Laughs 攻击：lol="lol"(3字符) → lol2=&lol;×10(30字符) → ... → lol9=3亿字符！解析器试图展开3亿字符 → 内存耗尽 → DoS。
4

CSRF 缺失

中危

CWE-352 · XML 导入接口无 CSRF Token 校验

CSRF + XXE = 跨站文件读取：攻击者构造恶意页面诱导管理员访问，浏览器自动提交恶意 XML。

5

错误信息泄露

低危

修复前：error = f"解析失败: {e}" 暴露文件路径、文件存在性、权限信息。

三、XXE 防护方案

Security Hardening

3.1 最佳策略：根本不用 XML 解析器

策略1：禁用 XML 解析器外部实体 ★★★★★ 策略2：使用正则代替 XML 解析 ★★★★★（本方案） 没有解析器 → 没有实体解析 → 没有 XXE 策略3：禁用 DOCTYPE ★★★★ 策略4：过滤 ENTITY/SYSTEM 关键字 ★★★（黑名单易绕过）

3.2 修复后代码

Python — 修复后
@app.route("/xml-import", methods=["GET", "POST"])
@login_required
def xml_import():
    if request.method == "POST":
        if not validate_csrf(): abort(403)

        xml_data = request.form.get("xml_data", "")
        if xml_data.strip():
            # 第1关：检测并拒绝 XXE 攻击
            if re.search(r'<!DOCTYPE.*\[', xml_data, re.IGNORECASE | re.DOTALL):
                error = "不支持包含 DOCTYPE 声明的 XML，禁止外部实体加载"
                return render_template("xml_import.html", error=error)

            # 第2关：正则直接提取 user 节点（不经过 XML 解析器）
            users = re.findall(
                r'\s*([^<]+)\s*([^<]+)\s*',
                xml_data
            )
            result = json.dumps([{"name":n,"email":e} for n,e in users],
                                indent=2, ensure_ascii=False)

    return render_template("xml_import.html", result=result, error=error,
                           csrf_token=generate_csrf_token())

3.3 修复原理

修复前：XML → 提取 ENTITY → open() 读文件 → 替换实体 → 信息泄露 ❌ 修复后：XML → 检测 DOCTYPE → 拒绝（"不支持"）→ 正则直接提取 ✅ 没有任何 XML 解析逻辑！没有实体解析、没有文件读取！

3.4 修复前后对比

代码段修复前修复后
CSRF 校验无validate_csrf()
DOCTYPE 检测无re.search(r'<!DOCTYPE.*\[', ...)
ENTITY 处理提取路径 + open() 读取删除
文件读取with open(path)删除
错误信息f"解析失败: {e}"统一提示

3.5 验证结果

1. 正常 XML 解析 ✅ 正确提取 name/email 2. XXE file:///etc/passwd ✅ 拦截（"不支持 DOCTYPE"） 3. XXE file:///app.py ✅ 拦截 4. 无 CSRF Token ✅ HTTP 403 5. 错误信息泄露 ✅ 统一提示

"最好的安全措施，不是让攻击者难以攻击，而是让攻击面根本不存在。"

XXE 漏洞的存在，本质上是由于使用了功能过剩的 XML 解析器。对于只需要提取几个字段的场景，XML 解析器的实体解析、DTD 处理、命名空间支持等功能全部是多余的攻击面。本轮的修复方案——用正则提取替代 XML 解析器——体现了一个重要的安全设计理念：去掉不需要的功能，就是最好的安全加固。
