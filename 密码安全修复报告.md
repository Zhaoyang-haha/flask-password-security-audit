# 密码安全修复报告

## Flask 用户管理系统 — 安全漏洞分析与修复

---

## 一、项目概述

本项目是一个基于 Python Flask 框架的简易用户信息管理平台，包含登录、用户信息展示、登出等基本功能。本次作业的目标是：**识别代码中的密码安全漏洞，并逐一修复**。

### 项目结构

```
flask_user_app/
├── app.py                 # Flask 主应用
├── templates/
│   ├── base.html          # 基础模板
│   ├── index.html         # 首页（用户信息展示）
│   └── login.html         # 登录页
└── static/
    └── css/
        └── style.css      # 样式文件
```

## 三、原始漏洞分析

### 漏洞 1：密码明文存储

**问题代码（`app.py`）：**

```python
USERS = {
    "admin": {
        "username": "admin",
        "password": "admin123",      # ← 明文存储！
        "role": "admin",
        "email": "admin@example.com",
        "phone": "13800138000",
        "balance": 99999
    },
    "alice": {
        "username": "alice",
        "password": "alice2025",     # ← 明文存储！
        "role": "user",
        "email": "alice@example.com",
        "phone": "13900139001",
        "balance": 100
    }
}
```

**危害：** 如果数据库文件被泄露（或攻击者通过其他漏洞获取了 USERS 字典的内容），所有用户的密码将以可读形式直接暴露。

**验证方式：** 查看源代码即可看到所有账号密码。

### 漏洞 2：密码明文比对

**问题代码（`app.py`）：**

```python
if username in USERS and USERS[username]["password"] == password:
```

**危害：** 使用 `==` 直接比字符串，不仅不安全（无法防御哈希碰撞），而且意味着密码必须明文存储才能比对。

### 漏洞 3：密码显示在页面上

**问题代码（`templates/index.html`）：**

```html
<li><span class="info-label">密码：</span>{{ user_info["password"] }}</li>
```

**危害：** 登录成功后，用户的密码直接渲染在网页上。如果用户离开电脑未锁屏，路过的人可直接看到密码。

**验证方式：** 登录后页面直接显示"密码：admin123"。

### 漏洞 4：HTML 注释泄露默认账号

**问题代码（`templates/login.html`）：**

```html
<!-- 调试信息 - 默认管理员账号 用户名: admin 密码: admin123 -->
```

**危害：** 任何查看网页源代码的人都能直接获得管理员账号密码。攻击者不需要进行任何爆破或猜测。

**验证方式：** 浏览器右键 → 查看页面源代码，即可看到注释中的账号密码。

### 漏洞 5：弱 Secret Key

**问题代码（`app.py`）：**

```python
app.secret_key = "dev-key-2025"
```

**危害：** `secret_key` 用于 Flask 签名 session cookie。使用硬编码的弱密钥，攻击者可以伪造任意用户的 session，实现**会话劫持**和**身份伪造**。

### 漏洞 6：Debug 模式开启

**问题代码（`app.py`）：**

```python
app.run(debug=True, host="0.0.0.0", port=5000)
```

**危害：** Flask 的 debug 模式会在代码出错时显示详细的调试页面，包含源代码片段、环境变量、当前变量值等敏感信息。生产环境中绝对不应开启。

### 漏洞 7：无爆破防护

**危害：** 没有登录失败次数限制，攻击者可以使用 Hydra 等工具对账号进行**无限次密码爆破**，直到猜中为止。

## 四、Burp Suite 抓包验证

### 抓包过程

1. 配置 Burp Suite 代理监听 `127.0.0.1:8080`
2. 浏览器设置代理为 `127.0.0.1:8080`
3. 访问 `http://192.168.14.129:5000/login`
4. 输入用户名 `admin`，密码 `admin123` 并提交
5. Burp Suite 拦截到 POST 请求，内容如下：

```
POST /login HTTP/1.1
Host: 192.168.14.129:5000
Content-Type: application/x-www-form-urlencoded

username=admin&password=admin123    ← 密码明文传输
```

### 验证截图说明

> Burp Suite 的 Proxy → Intercept 面板显示拦截到的 HTTP 请求，`username=admin&password=admin123` 以明文形式出现在请求体中。

## 五、修复方案

### 修复 1：使用 Werkzeug 进行密码哈希存储

**修复思路：** 不存储明文密码，而是存储密码的哈希值。验证时用同样的哈希算法比对。

**修复后代码（`app.py`）：**

```python
from werkzeug.security import generate_password_hash, check_password_hash

USERS = {
    "admin": {
        "username": "admin",
        "password": generate_password_hash("admin123"),  # ← 哈希存储
        # ...
    },
    "alice": {
        "username": "alice",
        "password": generate_password_hash("alice2025"), # ← 哈希存储
        # ...
    }
}
```

**效果：** 数据库中存储的是类似 `pbkdf2:sha256:600000$salt$hash` 的哈希值，即使数据泄露也无法还原出原始密码。

### 修复 2：使用 check_password_hash 进行密码验证

**修复后代码（`app.py`）：**

```python
# 修复前
if USERS[username]["password"] == password:

# 修复后
if check_password_hash(USERS[username]["password"], password):
```

**效果：** 使用 Werkzeug 提供的安全比对函数，防止时序攻击，同时兼容哈希存储。

### 修复 3：不在模板中输出密码

**修复后代码（`templates/index.html`）：**

```html
<!-- 修复前 -->
<li><span class="info-label">密码：</span>{{ user_info["password"] }}</li>

<!-- 修复后 --> ← 整行删除，不再显示密码
```

同时在 `app.py` 中过滤掉密码字段：

```python
# 传递给模板前过滤掉密码
user_info = {k: v for k, v in USERS[username].items() if k != "password"}
```

### 修复 4：删除泄露信息的 HTML 注释

**修复后代码（`templates/login.html`）：**

```html
<!-- 整行删除 -->
<!-- 调试信息 - 默认管理员账号 用户名: admin 密码: admin123 -->
```

### 修复 5：使用随机 Secret Key

**修复后代码（`app.py`）：**

```python
import os
app.secret_key = os.urandom(32)  # ← 每次启动随机生成
```

### 修复 6：关闭 Debug 模式

**修复后代码（`app.py`）：**

```python
app.run(debug=False, host="0.0.0.0", port=5000)
```

### 修复 7：添加爆破防护

**修复后代码（`app.py`）：**

```python
# 登录失败计数
login_attempts = {}

# 在登录验证中检查
client_ip = request.remote_addr
if client_ip in login_attempts:
    attempts, lock_time = login_attempts[client_ip]
    if attempts >= 3 and (time.time() - lock_time) < 60:
        error = "登录失败次数过多，请60秒后再试。"
        return render_template("login.html", error=error)

# 登录失败后记录
login_attempts[client_ip] = (count + 1, now)

# 登录成功后清除
login_attempts.pop(client_ip, None)
```

**效果：** 同一 IP 连续输错 3 次密码后锁定 60 秒，大幅增加爆破难度。

## 六、修复前后对比

| 漏洞项 | 修复前 | 修复后 |
|--------|--------|--------|
| 密码存储 | 明文 `"admin123"` | 哈希 `pbkdf2:sha256:...` |
| 密码比对 | `==` 字符串直接比 | `check_password_hash()` |
| 密码显示 | 页面上显示密码 | 密码不再出现在任何页面 |
| 默认账号泄露 | HTML 注释中写死 | 删除注释 |
| Secret Key | 硬编码 `"dev-key-2025"` | 随机生成 `os.urandom(32)` |
| Debug 模式 | 开启 `debug=True` | 关闭 `debug=False` |
| 爆破防护 | 无限制 | 3次失败锁定60秒 |
| HTTP 明文传输 | 未修复 | （需配置 HTTPS，本次未涉及） |

## 八、总结与反思

### 安全开发原则

1. **永远不要存储明文密码** — 使用经过充分测试的哈希库（如 `werkzeug.security`、`bcrypt`、`argon2`）
2. **最小权限原则** — 不应在前端展示敏感字段（密码、密钥等）
3. **不要信任注释** — 代码注释中的敏感信息同样是安全风险
4. **使用安全的默认配置** — `debug=True` 仅用于开发环境，上线前必须关闭
5. **防御纵深** — 密码哈希防泄露，爆破防护防暴力破解，HTTPS 防中间人攻击

### 本次作业未修复项

- **HTTP 明文传输：** 由于在内网测试环境，未配置 HTTPS。生产环境必须使用 HTTPS + HSTS 防止中间人窃听。
- **Session 配置：** 可进一步配置 `session_cookie_secure=True`、`session_cookie_httponly=True` 等安全选项。
- **CSRF 防护：** 登录表单未添加 CSRF Token，可被跨站请求伪造攻击利用。
