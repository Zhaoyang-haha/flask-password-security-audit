## 第一章：实验概述

### 1.1 实验目的

1. 理解**业务逻辑漏洞**[Business Logic Vulnerability]的产生原理与攻击手法
2. 理解**IDOR 越权访问**[Insecure Direct Object Reference]的产生原理与攻击手法
3. 掌握使用 Burp Suite 进行业务逻辑漏洞测试的方法
4. 学会识别不安全的业务实现中的 8 类典型逻辑漏洞
5. 掌握业务逻辑安全的**服务端校验原则**与**权限校验模型**
6. 建立"客户端不可信"的安全开发思维

### 1.2 什么是业务逻辑漏洞？

业务逻辑漏洞是指**攻击者利用应用程序业务规则中的缺陷**，通过非常规的请求顺序、参数值或操作方式，实现开发者未曾预期的行为。

| 特征 | 说明 |
|------|------|
| **非技术性** | 不是协议或加密层面的漏洞，而是"规则设计"有问题 |
| **业务相关** | 每个应用的业务逻辑不同，没有通用的扫描规则 |
| **难以检测** | 自动化扫描器难以发现，需要人工分析业务流程 |
| **危害巨大** | 可能直接导致金融损失或数据泄露 |

### 1.3 什么是 IDOR 越权访问？

IDOR[Insecure Direct Object Reference，不安全的直接对象引用]是指**应用程序在访问内部对象[如数据库记录、文件]时，直接使用了用户提供的标识符[如 ID、用户名]，但没有验证当前用户是否有权访问该对象**。

```
正常情况：
  用户A → 请求 profile?user_id=A → 服务器返回 A 的资料 ✅

IDOR 漏洞：
  用户A → 请求 profile?user_id=B → 服务器也返回 B 的资料 ❌
  [A 修改了 URL 中的 user_id 参数，就看到了 B 的隐私数据]
```

本轮在已有功能基础上新增了**个人中心**和**充值**两个模块，其中故意保留了多种业务逻辑与越权漏洞：

## 第二章：业务逻辑漏洞基础理论

### 2.1 业务逻辑漏洞的产生原因

```
业务逻辑漏洞 = 开发者的"信任假设" + 攻击者的"逆向思维"
```

常见的原因包括：

| 原因 | 说明 | 示例 |
|------|------|------|
| **信任客户端** | 认为用户不会修改请求参数 | 金额、数量、价格放在隐藏字段 |
| **只检查前端** | 只在浏览器端做 JS 校验 | 前端限制金额最小为0，后端不校验 |
| **忽略负值/边界** | 认为用户只会输入正数 | 数量 -1 导致账户余额增加 |
| **忽略流程顺序** | 认为用户会按顺序操作 | 跳过支付步骤直接确认订单 |
| **权限校验缺失** | 认为用户不会修改 ID 参数 | URL 中的 user_id 未验证归属 |

### 2.2 客户端不可信原则

```
┌─────────────────────────────────────────────┐
│              安全开发第一定律                   │
│                                               │
│   永远不要信任客户端传来的任何数据！             │
│                                               │
│   客户端包括：                                 │
│   · 浏览器[用户可修改 HTML/JS]               │
│   · 表单参数[用户可修改 POST 数据]            │
│   · URL 参数[用户可任意修改]                  │
│   · Cookie[用户可修改]                       │
│   · HTTP 请求头[用户可任意设置]               │
└─────────────────────────────────────────────┘
```

### 2.3 业务逻辑漏洞与常规漏洞的区别

| 对比维度 | 常规漏洞[如 SQL 注入] | 业务逻辑漏洞 |
|---------|----------------------|-------------|
| **原理** | 技术层面绕过安全机制 | 业务规则设计缺陷 |
| **检测** | 自动化扫描器可发现 | 需要人工分析业务流 |
| **通用性** | 跨应用通用 | 每个应用各不相同 |
| **危害** | 数据泄露/系统控制 | 金融损失/逻辑混乱 |
| **修复** | 加过滤/参数化查询 | 重新设计业务流程 |

## 第三章：IDOR 越权访问基础理论

### 3.1 什么是水平越权和垂直越权？

```
              ┌──────────────────┐
              │    管理员权限      │
              │   role = admin    │
              └────────┬─────────┘
                       │
          ┌────────────┼────────────┐
          │ 垂直越权    │            │
          ▼            ▼            ▼
     ┌──────────┐ ┌──────────┐ ┌──────────┐
     │  用户 A   │ │  用户 B   │ │  用户 C   │
     │ role=user │ │ role=user │ │ role=user │
     └──────────┘ └──────────┘ └──────────┘
          │              │
          └──────┬───────┘
                 │ 水平越权
                 ▼
          用户 A 访问用户 B 的资料
```

| 越权类型 | 说明 | 示例 |
|---------|------|------|
| **水平越权** | 同级别用户之间互相访问 | 普通用户 A 查看普通用户 B 的资料 |
| **垂直越权** | 低权限用户访问高权限功能 | 普通用户执行管理员操作 |

### 3.2 IDOR 的常见场景

```
场景1：URL 中的数字 ID
  /profile?user_id=123 → 修改为 124 就能看别人的资料

场景2：文件名中的 ID
  /download/invoice_123.pdf → 修改为 124 就能下载别人的发票

场景3：API 中的 UUID
  /api/user/abc-def → 如果 UUID 可预测或枚举，导致数据泄露

场景4：数字账户操作
  /transfer?from=123&to=456 → 修改 from 参数从他人账户转出
```

### 3.3 IDOR 的防御原则

```
防御 IDOR 的三种方法[按推荐程度排序]：

方法1：基于 Session 获取身份 ★★★★[最推荐]
  不从客户端参数获取 user_id，而是从 session 中获取
  代码：user_id = session["user_id"]
  优点：用户无法修改，天然防御 IDOR

方法2：验证对象归属[次推荐]
  从客户端获取 user_id，但验证该 ID 是否属于当前用户
  代码：if user_id != session["user_id"]: abort(403)

方法3：使用不可预测的标识符[辅助]
  用 UUID 替代自增数字 ID
  限制：不能替代归属验证，只能防止枚举
```

## 第五章：漏洞一 -- IDOR 越权查看他人资料

### 5.1 漏洞信息

| 项目 | 内容 |
|------|------|
| 漏洞类型 | IDOR[Insecure Direct Object Reference]-- 水平越权 |
| 危害等级 | **严重[Critical]** -- CVSS 8.6 |
| CWE 编号 | CWE-639: Authorization Bypass Through User-Controlled Key |
| OWASP Top 10 | A01:2021 – Broken Access Control |
| 影响功能 | 个人中心 `GET /profile?user_id=N` |

### 5.2 漏洞代码分析[修复前]

```python
@app.route("/profile")
@login_required
def profile():
    user_id = request.args.get("user_id", "")   # 从 URL 参数获取 -- 用户可控！
    ...
    sql = f"SELECT ... FROM users WHERE id = {user_id}"
    c.execute(sql)
    ...
    return render_template("profile.html", user=user_data)
```

**根本问题：** 从 `request.args`[URL 参数]获取 `user_id`，用户修改 URL 即可访问任意用户的资料。完全没有验证当前登录用户和被查询用户之间的关系。

**修复后[正确做法]：**

```python
@app.route("/profile")
@login_required
def profile():
    # 不从 URL 参数获取 user_id，而是从 session 推导
    current_user_id = get_current_user_id()     # 基于 session 中的 username
    ...
    c.execute("SELECT ... FROM users WHERE id = ?", (current_user_id,))
```

### 5.3 攻击复现 -- POC 1

**攻击目标：** 普通用户通过修改 URL 参数查看管理员或其他用户的资料。

**前提条件：** 任意用户登录系统。

**攻击步骤：**

```bash
# 步骤1：以普通用户 alice 身份登录
curl -c /tmp/alice.txt -X POST http://192.168.14.129:5000/login \
  -d "username=alice&password=alice2025"

# 步骤2：查看自己的资料[正常]
curl -b /tmp/alice.txt "http://192.168.14.129:5000/profile?user_id=2"
# 返回：alice 的资料

# 步骤3：修改 user_id 查看管理员[越权！]
curl -b /tmp/alice.txt "http://192.168.14.129:5000/profile?user_id=1"
# 返回：admin 的资料[邮箱、手机、余额]-- 越权成功！

# 步骤4：遍历 user_id 查看所有用户
for id in 1 2 3 4 5; do
  curl -b /tmp/alice.txt "http://192.168.14.129:5000/profile?user_id=$id"
done
# 返回：所有用户的敏感信息 -- 批量数据泄露！
```

**攻击结果[修复前]：**

```
user_id=1 → 管理员资料          ✅ 越权成功
user_id=2 → alice 资料          ✅ 正常
user_id=3 → 注册用户资料        ✅ 越权成功
...
user_id=N → 任意用户资料        ✅ 越权成功
```

### 5.4 Burp Suite 测试过程

```
步骤1：拦截请求
  进入个人中心页面时，Burp 拦截到：
  GET /profile?user_id=2 HTTP/1.1
  Cookie: session=eyJ1c2VybmFtZSI6ImFsaWNlIn0...

步骤4：观察响应
  返回了 admin 用户的邮箱、手机、余额信息
  证明越权成功，管理员资料泄露
```

## 第六章：漏洞二 -- IDOR 越权为他人充值

### 6.1 漏洞信息

| 项目 | 内容 |
|------|------|
| 漏洞类型 | IDOR 越权操作 -- 水平越权 |
| 危害等级 | **严重[Critical]** -- CVSS 9.0 |
| CWE 编号 | CWE-639: Authorization Bypass Through User-Controlled Key |
| OWASP Top 10 | A01:2021 – Broken Access Control |
| 影响功能 | 充值接口 `POST /recharge` |

### 6.2 漏洞代码分析[修复前]

```python
@app.route("/recharge", methods=["POST"])
@login_required
def recharge():
    user_id = request.form.get("user_id", "")   # 从表单参数获取
    amount = request.form.get("amount", "0")
    ...
    sql = f"UPDATE users SET balance = balance + {amount} WHERE id = {user_id}"
```

**根本问题：** 充值操作的 `user_id` 来自表单[用户可控]，攻击者可以修改该参数为其他用户的 ID，实现**为他人充值**或**将他人余额转出**。

**修复后：**

```python
@app.route("/recharge", methods=["POST"])
@login_required
def recharge():
    ...
    current_id = get_current_user_id()     # 从 session 获取当前用户
    if not current_id or str(current_id) != str(user_id):
        abort(403, "无权操作其他用户的账户")   # 校验归属
```

### 6.3 攻击复现 -- POC 2

**攻击目标：** 低余额用户给管理员充值，或通过负值为他人扣款。

```bash
# 场景1：用户 alice 给 admin 充值
# Alice 登录后，正常充值请求：
# POST /recharge  user_id=2&amount=100

# 攻击：修改 user_id=1，给 admin 充值
curl -b /tmp/alice.txt -X POST http://192.168.14.129:5000/recharge \
  -d "user_id=1&amount=100"
# admin 的余额增加了 100 -- Alice 给 admin 充了钱

# 场景2：结合负值，从 admin 账户扣钱
curl -b /tmp/alice.txt -X POST http://192.168.14.129:5000/recharge \
  -d "user_id=1&amount=-10000"
# admin 的余额减少了 10000 -- 从 admin 账户窃取了余额！
```

### 6.4 攻击危害分析

| 攻击者 | 操作 | 效果 | 危害程度 |
|--------|------|------|---------|
| 普通用户 | 修改 `user_id=1`，正常充值 | 给 admin 加钱[看似无害] | 低 |
| 普通用户 | 修改 `user_id=1`，大额充值 | 让 admin 有巨额"不明资金" | 中 |
| 普通用户 | 修改 `user_id=2`[自己]，超大额 | 给自己无限刷钱 | 高 |
| 普通用户 | 修改 `user_id=1`，负值大额 | 从 admin 账户扣钱到自己账户 | **严重** |

## 第七章：漏洞三 -- 负值金额充值[业务逻辑缺陷]

### 7.1 漏洞信息

| 项目 | 内容 |
|------|------|
| 漏洞类型 | 业务逻辑缺陷 -- 金额校验缺失 |
| 危害等级 | **高危[High]** -- CVSS 7.5 |
| CWE 编号 | CWE-841: Improper Enforcement of Behavioral Workflow |
| 影响功能 | 充值接口 `POST /recharge` |

### 7.2 漏洞代码分析[修复前]

```python
amount = request.form.get("amount", "0")
# 未对 amount 做任何正负校验！
amount = float(amount)
sql = f"UPDATE users SET balance = balance + {amount} WHERE id = {user_id}"
```

**SQL 语义分析：**

```
正常充值[amount = 100]：
  UPDATE users SET balance = balance + 100    → 余额增加 100

恶意充值[amount = -100]：
  UPDATE users SET balance = balance + (-100)  → 余额减少 100

恶意提现[amount = -99999]：
  UPDATE users SET balance = balance + (-99999) → 余额几乎清零
```

### 7.3 攻击复现 -- POC 3

```bash
# Alice 查看自己余额
```

## 第十章：漏洞六 -- SQL 注入[个人中心与充值接口]

### 10.1 漏洞信息

| 项目 | 内容 |
|------|------|
| 漏洞类型 | SQL 注入[SQL Injection] |
| 危害等级 | **严重[Critical]** -- CVSS 9.8 |
| 影响功能 | 个人中心 `GET /profile` + 充值 `POST /recharge` |

### 10.2 漏洞代码分析[修复前]

```python
# 个人中心 -- f-string 拼接
sql = f"SELECT id, username, email, phone, balance FROM users WHERE id = {user_id}"
c.execute(sql)

# 充值 -- f-string 拼接
sql = f"UPDATE users SET balance = balance + {amount} WHERE id = {user_id}"

```

**注入点分析：**

```
个人中心的注入点[user_id 参数]：
  正常：/profile?user_id=1
  注入：/profile?user_id=1 UNION SELECT 1,2,3,4,5--

  结果：返回 UNION 注入的结果

  正常：amount=100
  注入：amount=100; UPDATE users SET balance=99999 WHERE id=2--
  生成 SQL：UPDATE users SET balance = balance + 100; UPDATE users SET balance=99999 WHERE id=2-- WHERE id = 1
  结果：其他用户的余额被篡改
```

### 10.3 修复方案

```python

    "SELECT id, username, email, phone, balance FROM users WHERE id = ?",
```

### 12.2 漏洞分析[修复前]

```python
    session["csrf_token"] = secrets.token_hex(16)
```

## 第十四章：业务逻辑漏洞防护方案

### 14.1 权限校验模型

```
正确的权限校验模型：

              客户端请求
                  │
                  ▼
          ┌─────────────────┐
          │   身份认证       │  ← 是否已登录？[Session 校验]
          │   @login_required│
          └────────┬────────┘
                   │
          ┌────────▼────────┐
          │   对象归属验证    │  ← 操作的对象是否属于当前用户？
          │   user_id = self │     不是从参数获取，而是从 Session 获取
          │   → 水平越权防护 │
          └────────┬────────┘
                   │
          ┌────────▼────────┐
          │   业务参数校验    │  ← 金额、数量是否在合理范围内？
          │   amount > 0     │     防止负值、超额、零值
          │   → 逻辑漏洞防护 │
          └────────┬────────┘
                   │
          ┌────────▼────────┐
          │   CSRF Token校验 │  ← 请求是否是用户本意？
          │   validate_csrf │     防止跨站伪造
          │   → CSRF 防护   │
          └────────┬────────┘
                   │
          ┌────────▼────────┐
          │   防重放         │  ← 是否重复提交？
          │   刷新 CSRF Token│     防止重放攻击
          │   → 重放防护    │
          └────────┬────────┘
                   │
                   ▼
              正常执行业务逻辑
```

### 14.2 金额操作安全规范

| 规范 | 说明 | 绕过方式 | 防御措施 |
|------|------|---------|---------|
| 正数校验 | 金额必须大于0 | 输入负数 | `if amount <= 0: reject` |
| 上限校验 | 金额不能过大 | 输入极大值 | `if amount > 100000: reject` |
| 精度校验 | 防止浮点精度问题 | 输入 0.001 分 | 限制小数位数 |
| 身份绑定 | 只能操作自己的账户 | 修改 user_id | 从 Session 获取 ID |
| 防重放 | 同一请求不能重复执行 | 重复提交 | CSRF Token 一次性使用 |
| 频率限制 | 防止批量攻击 | 快速提交 | 接口限速 |

### 14.3 完整的修复代码

```python
# ===== 个人中心[修复后]=====

@app.route("/profile")
@login_required
def profile():
    user_data = None
    error = request.args.get("error", "")
    current_user_id = get_current_user_id()    # 从 Session 获取，不支持外部修改

    if current_user_id is not None:
        conn = get_db()
        c = conn.cursor()
        # 参数化查询
        c.execute(
            "SELECT id, username, email, phone, balance FROM users WHERE id = ?",
            (current_user_id,),
        )
        row = c.fetchone()
        conn.close()
        if row:
            user_data = {
                "id": row["id"],
                "username": row["username"],
                "email": row["email"] or "",
                "phone": row["phone"] or "",
                "balance": row["balance"] or 0,
            }

    return render_template("profile.html", user=user_data, csrf_token=generate_csrf_token(), error=error)


# ===== 充值[修复后]=====

@app.route("/recharge", methods=["POST"])
@login_required
def recharge():
    # 1. CSRF 校验
    if not validate_csrf():
        abort(403, "CSRF 验证失败")

    user_id = request.form.get("user_id", "")
    amount_str = request.form.get("amount", "0")

    # 2. IDOR 校验：只能操作自己的账户
    current_id = get_current_user_id()
    if not current_id or str(current_id) != str(user_id):
        abort(403, "无权操作其他用户的账户")

    # 3. 金额格式校验
    try:
        amount = float(amount_str)
    except ValueError:
        return redirect("/profile?error=金额格式不正确")

    # 4. 金额正数校验
    if amount <= 0:
        return redirect("/profile?error=充值金额必须大于0")

    # 5. 金额上限校验
    if amount > 100000:
        return redirect("/profile?error=单次充值金额不能超过10万元")

    # 6. 参数化查询
    conn = get_db()
    c = conn.cursor()
    c.execute(
        "UPDATE users SET balance = balance + ? WHERE id = ?",
        (amount, user_id),
    )
    conn.commit()
    conn.close()

    # 7. 刷新 CSRF Token[防重放]
    refresh_csrf()

    return redirect("/profile")
```

### 14.4 辅助函数说明

```python
def get_current_user_id():
    """基于 Session 获取当前用户 ID[关键安全函数]"""
    username = session.get("username", "")
    if not username:
        return None
    conn = get_db()
    c = conn.cursor()
    c.execute("SELECT id FROM users WHERE username = ?", (username,))
    row = c.fetchone()
    conn.close()
    return row["id"] if row else None
```

```bash
HTTP1=$(curl -s -o /dev/null -w "%{http_code}" "$BASE/profile?user_id=2" -b "$COOKIE")
echo "访问 ?user_id=2[admin登录]: HTTP $HTTP1[应200，但不显示alice数据]"

echo ""
echo "测试2：负值充值"
echo "---------------"
P_CSRF=$(curl -s "$BASE/profile" -b "$COOKIE" | grep -oP 'csrf_token" value="\K[a-f0-9]+')
curl -s -o /dev/null -X POST "$BASE/recharge" -b "$COOKIE" \
  -d "csrf_token=$P_CSRF&user_id=1&amount=-500" -w "状态: %{http_code}\n"

echo ""
echo "测试3：CSRF 缺失"
echo "-----------------"
curl -s -o /dev/null -w "状态: %{http_code}\n" -X POST "$BASE/recharge" \
  -b "$COOKIE" -d "user_id=1&amount=100"

echo ""
echo "测试4：越权充值他人"
echo "-------------------"
P2=$(curl -s "$BASE/profile" -b "$COOKIE")
P2_CSRF=$(echo "$P2" | grep -oP 'csrf_token" value="\K[a-f0-9]+')
curl -s -o /dev/null -w "状态: %{http_code}\n" -X POST "$BASE/recharge" \
  -b "$COOKIE" -d "csrf_token=$P2_CSRF&user_id=2&amount=100"

echo ""
echo "测试5：正常充值"
echo "---------------"
P3=$(curl -s "$BASE/profile" -b "$COOKIE")
P3_CSRF=$(echo "$P3" | grep -oP 'csrf_token" value="\K[a-f0-9]+')
curl -s -o /dev/null -w "状态: %{http_code}\n" -X POST "$BASE/recharge" \
  -b "$COOKIE" -d "csrf_token=$P3_CSRF&user_id=1&amount=50"

echo ""
echo "========================================="
echo " 验证完成"
echo "========================================="
```

## 第十七章：总结与反思

### 17.1 实验核心收获

通过本次业务逻辑漏洞与越权访问的分析与修复实践，掌握了以下核心知识：

**1. 业务逻辑安全的三大铁律**

```
铁律1：永远从 Session 获取当前用户身份，不信任客户端参数
铁律2：所有业务参数必须在服务端做范围校验[正/负、上限/下限]
铁律3：幂等操作必须防重放[一次性 Token + 唯一请求标识]
```

**2. IDOR 漏洞的三种防御层次**

```
最佳方案：完全避免从客户端获取对象标识符
         → 从 Session / Token 推导用户身份

次优方案：验证对象归属
         → 比较参数中的 ID 是否与当前用户匹配

兜底方案：使用不可预测的标识符
         → UUID 替代自增 ID[但不能替代归属验证]
```

**3. 本次修复的关键指标**

```
发现的业务逻辑/越权漏洞总数：   8 个
已修复的漏洞数：                8 个[修复率 100%]
涉及的安全机制：                6 种
  · Session 身份绑定
  · 金额正数校验
  · 金额上限校验
  · 参数化查询
  · CSRF Token 校验
  · CSRF Token 刷新[防重放]
```

### 17.2 8个漏洞回顾

| # | 漏洞 | 路由 | 严重程度 | 攻击方式 | 修复关键 |
|---|------|------|---------|---------|---------|
| V1 | IDOR 越权查看 | `/profile` | 严重 | 修改 `?user_id=N` | 从 Session 获取 ID |
| V2 | IDOR 越权充值 | `/recharge` | 严重 | 修改 `user_id` 表单值 | 比对 Session ID |
| V3 | 负值充值 | `/recharge` | 高危 | 输入负金额 | `amount <= 0` 拒绝 |
| V4 | 超额充值 | `/recharge` | 高危 | 输入极大金额 | `amount > 100000` 拒绝 |
| V5 | 重复提交 | `/recharge` | 中危 | 重复发送同请求 | 刷新 CSRF Token |
| V6 | SQL 注入 | `/profile`+`/recharge` | 严重 | 注入 payload 到 id/amount | 参数化查询 |
| V7 | CSRF 跨站 | `/recharge` | 中危 | 跨站提交表单 | `validate_csrf()` |
| V8 | 重放攻击 | `/recharge` | 中危 | 多次发送拦截的请求 | 一次性 Token |

### 17.3 安全开发检查清单

```
认证与会话管理：
[ ] 敏感操作是否从 Session 获取用户身份，而非客户端参数？
[ ] 是否验证了操作对象属于当前用户？
[ ] Session 是否配置了 HttpOnly + SameSite？

业务参数校验：
[ ] 金额、数量等数值是否做了正数校验？
[ ] 是否设置了合理的上限和下限？
[ ] 是否校验了数据类型[字符串/整数/浮点]？
[ ] 是否在校验前端没有限制的参数？

防篡改与防重放：
[ ] 关键业务操作是否有 CSRF 防护？
[ ] 关键业务的 CSRF Token 是否一次性使用？
[ ] 是否有操作频率限制？

数据库操作：
[ ] 所有 SQL 是否使用参数化查询[禁止拼接]？
[ ] 错误信息是否没有暴露 SQL 结构或数据？
```

### 17.4 后续学习方向

| 方向 | 说明 | 难度 |
|------|------|------|
| 竞争条件 | 并发请求导致数据不一致[如重复提现] | 进阶 |
| JWT 安全 | Token 伪造和签名算法混淆 | 进阶 |
| OAuth 2.0 安全 | 授权码劫持、CSRF、重定向 URI 篡改 | 高阶 |
| 支付逻辑 | 金额篡改、数量篡改、订单替换 | 高阶 |
| 业务流程绕过 | 跳过步骤、逆向操作、混合攻击 | 高阶 |

### 17.5 结束语

> **业务逻辑漏洞是所有安全漏洞中最"人性化"的一种。**
>
> 它不是因为程序员忘了加过滤，而是因为程序员**默认用户会按规矩操作**。但安全开发的核心思维恰恰是：**永远假设用户会尝试做任何你能想象到[甚至想象不到]的坏事。**
>
> 一个安全的业务系统 = 正确的功能逻辑 + 充分的校验逻辑 + 完整的权限控制
