Project Overview & Functionality

三 — 十、漏洞分析与修复

Vulnerability Analysis & Remediation

1

SQL 注入 — 搜索功能

严重 · CVSS 9.8

CWE-89 · 搜索接口 GET /search?keyword=xxx

Python — 问题代码
keyword = request.args.get("keyword", "")
sql = f"SELECT id, username, email, phone FROM users \
        WHERE username LIKE '%{keyword}%' OR email LIKE '%{keyword}%'"
c.execute(sql)

攻击复现：输入 ' OR '1'='1 窃取全部用户数据；' UNION SELECT 1,2,3,4-- 注入伪造数据。

❌ 修复前f"WHERE username LIKE '%{k}%'"
✅ 修复后"WHERE username LIKE ?" 参数绑定
2

SQL 注入 — 注册功能

严重 · CVSS 9.8

CWE-89 · 注册接口 POST /register（4 个字段均可注入）

Python — 问题代码
sql = f"INSERT INTO users (username, password, email, phone) \
        VALUES ('{username}', '{password}', '{email}', '{phone}')"
c.execute(sql)

攻击复现：用户名注入 hacker', 'pass', 'h@x.com', '123')-- 创建非法用户

更严重攻击场景：

  • '; DROP TABLE users; -- — 删除整个 users 表
  • '; UPDATE users SET password='hacked' WHERE username='admin'; -- — 修改管理员密码

修复：参数化查询 + 输入验证 + 密码哈希存储 + 格式校验

3

密码明文存储

高危 · CVSS 7.5

CWE-312 · SQLite 数据库 data/users.db

直接用 sqlite3 data/users.db "SELECT password FROM users" 即可看到 admin123、alice2025 等明文密码。

❌ 修复前明文存储 "admin123"
✅ 修复后generate_password_hash() 哈希存储

算法：Werkzeug scrypt（内存硬哈希），穷举 8 位密码约需 4 年（MD5 仅需 1 小时）。

4

错误信息泄漏数据库结构

中危

CWE-209 · 注册接口 POST /register

修复前：error = f"注册失败: {e}" 暴露 UNIQUE constraint failed: users.username，攻击者可推断数据库类型为 SQLite、存在 users 表、username 有 UNIQUE 约束、admin 是有效用户。

❌ 修复前f"注册失败: {e}"
✅ 修复后"该用户名已被注册" 统一提示
5

搜索接口未授权访问

中危

CWE-862 · 搜索接口 GET /search

修复前搜索接口无登录验证，未登录用户也可触发 SQL 查询（时间盲注风险）。

❌ 修复前无认证装饰器
✅ 修复后@login_required 装饰器
6

跨站请求伪造 CSRF

中危

CWE-352 · 登录 POST /login + 注册 POST /register

表单无 CSRF Token，攻击者可在恶意页面构造隐藏表单，受害者访问时自动提交请求。

修复： secrets.token_hex(16) 生成随机 Token 存入 Session，表单添加 <input type="hidden" name="csrf_token" value="{{ csrf_token }}">，提交时校验。
7

Session Cookie 安全配置缺失

中危

CWE-1004 · 全部页面

修复前 Cookie 仅有 Path=/，缺少 HttpOnly、SameSite、Secure 安全属性。

❌ 修复前Set-Cookie: session=...; Path=/
✅ 修复后HttpOnly; Path=/; SameSite=Lax
8

输入无长度限制

低危

全部输入点

修复前所有输入点均无长度和格式校验，存在数据库溢出和资源消耗风险。

修复：

Python
def validate_input(data, max_len=50):
    if not data or not data.strip():
        return False, "输入不能为空"
    if len(data) > max_len:
        return False, f"输入过长（最多{max_len}字符）"
    return True, ""

前端同步限制：<input maxlength="30">

十一、SQL 注入防护深度解析

SQL Injection Defense — Deep Dive

11.1 注入成功的三个条件

条件1：程序使用字符串拼接构建 SQL（f-string / %s / +）
条件2：用户输入包含 SQL 元字符（主要是单引号 '）
条件3：拼接后的 SQL 语法合法（攻击者构造出语法正确的语句）

11.2 参数化查询执行流程

步骤1：应用发送 SQL 模板 → "SELECT * FROM users WHERE username LIKE ?"
步骤2：数据库解析模板，生成查询执行计划（此时 SQL 结构已固定！）
步骤3：应用单独发送参数值 → "%' OR '1'='1%"
步骤4：数据库将参数填入执行计划，但不会重新解析为 SQL 代码
      实际效果：LIKE 匹配文本 "%' OR '1'='1%" — 这是字符串内容，不是 SQL 指令！

11.3 f-string 拼接 vs 参数化查询

步骤f-string 拼接参数化查询
第1步f"WHERE name = '{input}'" — 代码数据混合"WHERE name = ?" — 模板数据分离
第2步输入 ' OR '1'='1 成为 SQL 代码输入被当作参数值
第3步生成 WHERE name = '' OR '1'='1'数据库：这是参数，不是代码
结果⚠️ 注入成功✅ 注入失败

11.4 各编程语言参数化写法

语言驱动参数化（安全）危险写法
Pythonsqlite3c.execute("SELECT ?", (val,))f"SELECT '{val}'"
Pythonpsycopg2c.execute("SELECT %s", (val,))f"SELECT '{val}'"
JavaJDBCpstmt.setString(1, val)"SELECT '" + val + "'"
PHPPDO$stmt->execute([':v'=>$val])"SELECT '$val'"
Node.jsmysql2conn.query("SELECT ?", [val])\`SELECT '${val}'\`

11.5 多层防护体系

SQL 注入防护体系 第一层（最有效）：参数化查询 / 预编译语句 — 几乎 100% 防御 原理：数据与代码在数据库层面分离 第二层（辅助）：输入验证与过滤 — 作为第一层的补充 限制：容易被绕过（编码、大小写、注释符等） 第三层（辅助）：最小权限原则 效果：即使注入成功，也能限制损害范围 第四层（辅助）：Web 应用防火墙 WAF — 最后一道防线

十二、全部修复方案总结

Remediation Summary

修复前后对比总表

#漏洞名称严重修复前修复后
1SQL 注入（搜索）严重f"LIKE '%{k}%'""LIKE ?" 参数绑定

3密码明文存储高危INSERT ('admin123')generate_password_hash()

安全开发检查清单

[✓] 所有 SQL 操作使用参数化查询（? 占位符）
[✓] 完全禁止 f-string / % 拼接 / + 拼接 SQL
[✓] LIKE 子句中仍使用占位符
[✓] 错误信息不暴露 SQL 语法或数据库结构
[✓] 所有敏感接口有登录验证
[✓] 密码使用强哈希算法（scrypt）
[✓] 登录有防爆破机制
[✓] Session 配置 HttpOnly + SameSite
[✓] 所有用户输入有长度和格式校验
[✓] CSRF Token 保护所有 POST 请求
[✓] Debug 模式已关闭

核心收获

  • 不信任：永远不要信任用户的任何输入
  • 不拼接：永远不要用字符串拼接构建 SQL
  • 不泄露：永远不要在错误信息中暴露内部细节

安全不是一种功能，而是一种属性。 它不是在开发完成后"加"上去的，而是在每一行代码中"设计"出来的。

通过本次实验，从一个简单的 Flask 登录系统出发，经历了功能扩展（注册+搜索）、漏洞挖掘（8个漏洞）、安全加固（6种防护机制）的完整流程，建立了从"攻击者视角"思考问题、从"防御者视角"编码实践的双重视角。
